# 生成“编码任务计划说明书”

你的任务是把 `code-imitation-spec.md` 转化为一份真正能被新的 Coding Agent 直接执行的 `coding-task-plan.md`。这不是教学报告，也不是资料拼盘，而是一份完整、自包含、可直接发送给另一个 AI 的 Agentic Coding 提示词。

<HARD-GATE>
- 禁止输出真正的项目实现代码
- 禁止输出大段伪代码、完整代码片段或长代码块
- 必须优先使用自然语言描述类、属性、字段、方法、接口、状态、执行流程、边界条件、测试要求与实施顺序
- `coding-task-plan.md` 必须完全自包含，不允许依赖 `analysis-report.md`、`code-imitation-spec.md` 或当前对话补足核心上下文
- `coding-task-plan.md` 不得使用“参考库”“目标库”这类只在当前工作流中成立的元术语；必须直接写出具体项目路径、模块名、数据结构、流程与约束
- 采用单文件分章节门控：每完成一章都必须等待用户批准，未获批准不得继续下一章
</HARD-GATE>

## 全局提问原则

本工作流要求：**尽可能在计划阶段提出问题，好过于真正编码时再发现关键缺口。**

在本阶段必须遵守以下原则：

1. 在写某章之前，先检查当前规格与项目事实是否足以支持这一章可靠落笔。
2. 若发现会影响方向、文件落点、接口、模块职责、优先级、验收口径或具体实现路径的高影响不确定性，必须主动提问。
3. 若存在更优实现路线或更稳妥的落地方案，可以先提出方案建议与推荐，再继续写章节。
4. 提问应一次一个、聚焦决策，优先消除会导致后续编码质量下降或返工的问题。
5. 若可通过当前项目的轻量扫描确认事实，先探索；若属于用户意图或方案取舍，再问用户。

## 术语约定与阶段契约

- **参考库**：指被学习与提炼模式的来源，只在本技能内部用于整理知识；这些信息在最终文档中必须被改写成自包含事实，不能以“参考库”标签暴露给新的 AI。
- **目标库**：指本次编码任务真正要落地的代码库，只在本技能内部用于组织信息；最终文档必须直接写出具体仓库路径、模块名、约束与上下文，而不是使用“目标库”标签。
- **当前阶段产物**：指本阶段唯一要生成的 `coding-task-plan.md`。
- **输入**：一份已获用户批准的 `code-imitation-spec.md`，以及当前项目中可通过最小必要扫描确认的事实。
- **允许动作**：在规格基础上继续深入分析当前项目，补充更具体的文件落点、模块职责、类/字段/方法说明、接口与状态约束、执行流程、实施顺序、测试标准与工程约束；必要时做少量精确定位阅读、提问澄清高影响问题、提出更优落地方案。
- **禁止动作**：输出真正的项目实现代码、输出大段伪代码或完整代码片段、依赖外部文档补全关键信息、忽略当前项目集成边界、绕过章节级批准。
- **输出**：一份可单独交付的 `coding-task-plan.md`，整份文档就是可直接发送给新 AI 的完整提示词。
- **交接规则**：本计划必须依靠自身显式写出的内容接受审查；参考规格只用于核对一致性，不用于补齐缺失信息。

## 必须遵守的执行顺序

下面的顺序是强约束执行流，不是参考建议。若后文任何展开段落看起来允许跳步，以这里为准。

1. 读取已获用户批准的 `code-imitation-spec.md`
2. 只提取并消费下列显式字段：
   - 项目路径与相关模块 / 入口 / 配置 / 测试落点
   - 推荐方案与推荐理由
   - 目标库集成策略与模块落点
   - 关键数据、接口、状态与主流程约束
   - MVP 范围、非目标、成功标准与禁止事项
   - 风险、依赖与待确认事项
3. 先做规格完备性检查：
   - 若缺少会影响模块边界、文件落点、接口约束、实施顺序或验收口径的阻断性前提，必须回退到 `code-imitation-spec` 补齐
   - 只有非阻断性遗漏，才允许在当前项目最小必要扫描中补全
4. 基于规格做当前项目最小必要扫描，补齐更可实施的细节：
   - 相关目录、文件、类型、接口、配置、测试或基础设施的真实落点
   - 与推荐方案直接相关的现有模块职责、状态归属、协作边界与不能破坏的行为
   - 会影响实现顺序的依赖关系、迁移约束、调试抓手或验证入口
5. 把规格扩写成更具体的实现蓝图：
   - 需要新增 / 修改的文件、模块、类、服务、函数、类型或配置项
   - 每个核心对象的职责、关键字段 / 属性 / 类型、关键方法作用与执行流程
   - 主流程、边界流程、错误路径、状态变化与副作用边界
6. 确定 `coding-task-plan.md` 的章节顺序与分章策略
7. 生成当前章节时，必须：
   - 只使用自然语言描述实现要求，不输出大段伪代码或完整代码片段
   - 写清该章对应的文件 / 模块落点、职责、约束、流程与验收要求
   - 保证新的 AI 只看当前文档也能理解这一章要做什么
8. 每章写入同一个 `coding-task-plan.md` 文件后，暂停并等待用户批准
   - 若用户要求修改，先更新当前章，再重新等待批准
9. 继续下一章前，重新检查是否出现新的阻断性歧义；若有，先提问或提出方案，不要硬写
10. 全部章节通过后，派发 reviewer 做全文自审
11. 等待用户批准最终计划

## 产出目标

最终的 `coding-task-plan.md` 应帮助新的 Coding Agent 明确回答：

1. 要在当前项目中实现什么
2. 为什么做、做多少、不做什么
3. 应接到哪些现有模块或入口，复用或改写哪些现有能力
4. 关键模块、文件、类、方法、数据结构、接口契约、主流程与边界流程是什么
5. 实施顺序、测试要求与验收标准是什么
6. 哪些设计不能偏离，哪些复杂性暂时不要做

## 写作原则

- **项目内联**：整份文档必须直接写出项目路径、模块名、文件落点、接口约束与上下文，不得用“目标库”“参考库”代称。
- **自包含**：所有必要上下文都必须内联，不得出现“见参考报告”“见规格”“见前文”这类外部依赖表达。
- **在规格基础上深化**：`code-imitation-spec.md` 是基线，不是上限；你要补充更具体的项目内细节，而不是简单复述规格。
- **自然语言描述实现**：优先用自然语言写清类、字段、属性、方法、接口、状态、流程与错误处理；不要把空间浪费在大段伪代码或完整代码片段上。
- **细节充分**：如果缺少详细约束会让新的 AI 产出低质量代码，就必须把这些约束写进文档，而不是追求简短。
- **先澄清再落笔**：发现高影响不确定性时先提问或提方案，不得硬写。
- **软模板，不硬套标题**：章节标题、编号和拆分方式可以按任务复杂度调整；只要完整覆盖后文信息块，就不要机械复刻固定标题。

## 第一步：先把规格当成实现基线

在开始分章写作前，你必须先弄清：

- 规格已经确认了哪些边界、推荐方案、模块落点与核心约束
- 哪些内容仍然过于高层，需要在当前项目里再补一层具体事实
- 哪些风险、依赖、兼容或测试要求会直接影响实现顺序

## 第二步：在当前项目中补齐可实施的细节

你需要把规格中的高层决策继续压实到项目内部，至少补齐：

- 真实的目录、文件、模块、类型、接口、配置、测试与脚手架落点
- 哪些地方应复用，哪些地方应扩展，哪些地方应新增
- 哪些现有行为、契约、状态转换、生命周期或副作用边界不能破坏
- 哪些地方最容易踩坑、最需要验证、最需要加观察点

## 第三步：把规格扩写成详细实现蓝图

这里的“详细”不是贴代码，而是把接手实现的 AI 需要知道的实现信息写清楚，例如：

- 一个类：它应该放在哪个文件 / 模块、承担什么职责、有哪些关键属性 / 字段、每个字段的类型与语义是什么
- 一个方法：它负责什么、输入输出是什么、内部执行顺序是什么、依赖谁、可能返回什么错误或副作用
- 一个接口 / 类型：它表达什么约束、如何与现有结构映射、哪些字段是必填、哪些状态不变量不能破坏
- 一段流程：主链如何跑通、边界情况怎么处理、状态如何变化、哪里应该记录日志 / 暴露调试抓手

## 第四步：按章节生成并逐章门控

你写的不是“说明文”，而是另一个 AI 的执行提示词。因此：

- 每章都必须能回答“这一部分到底要改哪里、加什么、遵守什么约束、怎么判断完成”
- 每章完成后必须请求用户批准
- 未获批准不得继续下一章
- 若用户要求修改，先修当前章，不要跳章

## `coding-task-plan.md` 文档结构与章节要求

在进入文档结构要求之前，先明确最终文档契约：

- `coding-task-plan.md` 必须完全自包含；新的 AI 不应被要求回头看 `analysis-report.md`、`code-imitation-spec.md` 或当前对话。
- 文档必须明显比 `code-imitation-spec.md` 更具体：不仅有策略与边界，还要有项目内的文件 / 模块 / 类 / 方法 / 流程 / 测试层面的自然语言实现要求。
- 不允许输出大段伪代码或完整代码片段；如果需要表达结构，使用自然语言把字段名、类型、方法职责、执行顺序与错误处理写清。
- 文档可以很长；优先保证精确与完整，避免为了简短牺牲约束力。

`coding-task-plan.md` 的章节数可以随复杂度扩展，标题和编号也可以调整，但至少要覆盖下面这些信息块；如果某个主题复杂到一章无法写清，可以拆成两个连续章节。

### 信息块 1：任务定义与项目基线
必须写清：
- 当前项目的绝对路径或项目标识
- 本次要实现的能力与最终交付物
- 与本次任务最相关的现有模块、入口、基础设施与约束
- 本阶段明确不做的事情
- 成功标准与失败判定

### 信息块 2：已确认策略与现有代码上下文
必须写清：
- 从规格继承下来的推荐方案与核心理由
- 当前项目里已经存在什么能力可以复用
- 哪些模块 / 文件需要扩展，哪些模块 / 文件需要新增
- 哪些现有行为、契约、状态或流程不能破坏
- 与本次实现强相关的相邻环境，例如配置、测试、依赖集成、状态层、任务调度或边界模块

### 信息块 3：目标实现蓝图与模块 / 文件职责
必须写清：
- 建议模块划分与职责边界
- 模块之间的调用关系、状态归属与副作用边界
- 每个核心模块或文件在当前项目中的具体落点
- 对每个计划新增 / 修改的重要文件或模块，明确说明它的职责与与其他模块的协作关系

### 信息块 4：关键类型、类、方法与接口说明（自然语言）
必须写清：
- 关键数据结构、类型、字段、属性及其类型与语义
- 关键类 / 服务 / 函数 / 方法的名字、职责、输入、输出、依赖与副作用
- 关键状态转换规则、不变量与错误语义
- 新旧结构之间的映射关系（如果有）
- 全部用自然语言描述，不要写大段伪代码或完整代码片段

### 信息块 5：主流程、边界流程与状态变化
必须写清：
- 一条主执行链如何完整跑通
- 关键边界流程、错误路径、回退路径、幂等 / 重试 / 权限 / 兼容处理如何落地
- 关键状态如何变化，哪里应记录日志、暴露观察点或调试抓手
- 每条重要流程对应到哪些模块或方法负责

### 信息块 6：分阶段实施计划
必须写清：
- 推荐实施顺序，按依赖关系分步推进
- 每一步应该先修改哪些文件、模块、类型或配置
- 每一步的完成标志与进入下一步前的检查点
- 如果涉及迁移、兼容或发布顺序，必须写清先后关系

### 信息块 7：测试、验证与验收
必须写清：
- 最低测试要求、关键场景与验收方式
- 自动化测试、手动验证、观测或排障检查点
- 哪些失败结果算未完成
- 哪些行为必须保持不变

### 信息块 8：执行 Guardrails
必须写清：
- 新的 AI 在实现时必须遵守的约束
- 明确禁止事项与常见跑偏方式
- 遇到信息不足时应优先检查什么、什么时候应该停下来提问
- 哪些复杂性暂时不要做，避免过度实现

## 章节门控要求

- 每章完成后都必须请求用户批准。
- 未获批准不得生成下一章。
- 若用户要求修改，先更新当前章，再重新等待批准。
- 继续下一章前，重新检查当前文档是否还存在阻断性歧义。
- 全文通过后才能派发全文 reviewer 自审。

## 自审要求

reviewer 必须支持两级审查：

1. **章节级审查**：当前章节是否足够具体，是否还存在应当前置澄清的问题，是否已经把本章对应的文件 / 模块 / 类型 / 流程要求写清。
2. **全文级审查**：整份提示词是否自包含、是否完全服务当前项目、是否无外部依赖、是否比规格明显更具体、是否存在大段伪代码 / 代码片段、是否存在更优方案被忽略。

## Red Flags

- 整份文档仍然像教学报告，而不是执行说明
- 仍依赖上游文档补充关键上下文
- 文档继续使用“参考库”“目标库”这类新 AI 无法理解的工作流元术语
- 没有说明具体模块、文件、类、方法、接口和代码落点
- 关键数据结构、接口契约、主流程、边界流程或验收标准缺失
- 出现大段伪代码、完整代码片段或长代码块
- 明知有重大歧义，却继续硬写下一章
