# 设计“代码仿写需求规格说明书”

你的任务不是直接生成编码提示词，而是先把“从参考库学习什么、在目标库中落到哪里、哪些复杂性必须保留、哪些复杂性可以后置、哪些接口与流程约束已经足够明确”设计清楚，形成一份可审查、可交接、可落地的规格说明书。

<HARD-GATE>
此阶段只生成 `code-imitation-spec.md`，禁止直接生成 `coding-task-plan.md`，也禁止输出真正的项目实现代码。
</HARD-GATE>

## 全局提问原则

本工作流要求：**尽可能在规格阶段提出问题，好过于把关键歧义拖到后续实现规划或真正编码时再发现。**

在本阶段必须遵守以下原则：

1. 能在产出规格之前确认的信息，不要拖到后续实现规划时再发现。
2. 遇到影响目标库落点、能力边界、模块职责、数据契约、接口设计、集成方式、兼容策略、测试口径或验收标准的高影响不确定性时，必须优先提问。
3. 提问应一次一个、聚焦决策，优先消除会导致后续返工的问题。
4. 若可通过轻量扫描、目录检索、README、配置文件、类型定义、目标库入口代码或测试入口确认事实，先探索；若属于用户意图、取舍或优先级，再问用户。
5. 若 reviewer 发现规格仍缺少关键前提，应回到澄清，而不是硬把关键缺口留到后续实现规划。

## 术语约定与阶段契约

- **参考库**：指此前由 `clarify-and-plan` 与 `execute-analysis` 分析和学习的代码库，是本次仿写要学习模式、结构和约束的来源。
- **目标库**：指用户希望落地仿写结果的本地项目。目标库不是参考库的别名，而是一个需要被单独理解其技术栈、现有功能和集成边界的项目。
- **当前阶段产物**：指本阶段唯一要生成的 `code-imitation-spec.md`。
- **输入**：一份已获用户批准的 `analysis-report.md`，以及目标库路径、相关功能、技术栈与集成约束等显式信息。
- **允许动作**：提取分析报告中的显式发现，澄清目标库信息，进入目标库做最小必要调研，比较参考库与目标库差异，讨论候选落地方案，并定义仿写边界、模块落点、接口/状态约束、MVP、非目标、集成策略与风险。
- **禁止动作**：在缺少目标库关键信息时直接生成规格、把目标库默认成与参考库相同、直接生成 `coding-task-plan.md` 正文、依赖分析阶段的隐含上下文补齐规格缺口。
- **输出**：一份自包含、可继续细化的 `code-imitation-spec.md`，至少要写清双库背景、差异对照、目标库集成策略、模块落点、关键接口/状态约束、MVP 范围、必须保留的本质、允许简化项、明确不做项、风险与待确认事项。
- **交接规则**：后续使用者必须把本规格文档中显式写出的内容当作可信输入，不应默认知道分析报告的章节设计、作者脑中的隐含结构，或目标库的额外背景；后续工作可以继续细化这些输入，但不应重新决定本阶段已经收敛好的边界、推荐方案、模块落点与关键约束。

## 必须遵守的执行顺序

下面的顺序是强约束执行流，不是参考建议。若后文任何展开段落看起来允许跳步，以这里为准。

1. 读取已获用户批准的 `analysis-report.md`
2. 只提取并消费参考库的显式字段：
   - 技术栈与运行环境
   - 关键骨架、关键执行链与关键数据/状态模型
   - 可迁移本质与上下文限制
   - 明显受参考库上下文约束的部分
   - 报告中已经沉淀出的坑点、调试线索与适配提醒
3. 收集并验证目标库信息；至少覆盖：
   - 目标库路径或项目地址
   - 与待仿写能力最相关的模块、入口、配置、状态层、依赖集成点或测试入口
   - 用户希望在目标库中实现的能力边界与使用场景
   - 目标库当前技术栈、运行时、框架、关键基础设施与现实约束
   - 可能影响落地的兼容、迁移、发布、权限、性能、测试或观察性要求
4. 如果目标库信息不足，先发起澄清；必要时进入目标库做最小必要调研
5. 比较参考库与目标库在技术栈、架构分层、功能现状、状态管理、依赖集成、测试方式、扩展点与运行约束上的差异
6. 提出 `2-3` 种可行落地方案，写清 trade-off，并给出推荐方案
7. 向用户总结并确认：本次要迁移的能力边界、目标库落点、明确包含 / 不包含什么、推荐方案与推荐理由
   - 若用户调整边界、落点或方案偏好，回到第 3、4、5、6 步继续收敛
8. 基于已确认的信息，定义目标库集成策略、模块边界、关键接口/状态约束、MVP、非目标与风险
9. 按下方唯一写作入口生成 `code-imitation-spec.md`
10. 派发 reviewer 自审
11. 等待用户批准

## 核心理念

- 仿写不是“复制参考库总结”，而是“提炼参考库中值得迁移的本质，并把它们变成适合目标库落地的明确规格”。
- 此阶段起到承上启下的作用：上承参考库分析结论，下接目标库编码任务计划。
- `code-imitation-spec.md` 不是最终编码提示词，但它必须足够具体，才能让后续实现规划在此基础上继续深化，而不是重新发明边界与方案。

## 第一步：先提取参考库里真正要迁移的本质

规格中必须明确：

- **参考库中必须保留的本质**：哪些结构、流程、数据流、模块关系、接口假设或设计约束不能丢
- **明显受参考库上下文限制的部分**：哪些实现细节与技术选型不应被机械照搬
- **调试与演进经验**：报告里已经识别出的坑点、调试抓手、边界条件与扩展约束

## 第二步：一次一个问题，把双库迁移边界收敛到可写规格

### 澄清问题规则

- 一次只问一个问题
- 优先使用 A/B/C/D 多选；开放式问题也可以接受
- 重点围绕下面这些维度收集信息：
  - 目标能力边界与使用场景
  - 目标库中的具体落点与相邻模块
  - 关键数据结构、接口、状态或事件约束
  - 运行时、基础设施、权限、兼容或发布约束
  - MVP、非目标、测试与验收口径
- 如果问题已经能通过目标库轻量扫描确认，就不要再问用户
- 当下列信息已稳定时立即停止提问，进入方案设计：
  - 目标库路径与相关入口已明确
  - 本次要迁移的能力边界已明确
  - 关键模块落点与集成方向已基本稳定
  - 主要约束、非目标与验收口径已可落文

## 第三步：提出 `2-3` 种落地方案，并给出推荐

在规格成形之前，你必须给出 `2-3` 种可行落地方案，说明 trade-off，并明确推荐路线。

提出方案时必须写清：

- 每种方案准备如何把参考库本质接入目标库
- 每种方案最适合什么边界与约束
- 每种方案会牺牲什么、增加什么复杂性
- 推荐哪种方案，以及为什么它最适合当前目标库现实

## 第四步：先向用户确认目标边界、落点与推荐方案

在进入规格写作前，你必须先向用户总结并确认以下内容：

1. 本次准备迁移什么能力，以及解决什么问题
2. 目标库中准备接入哪些模块、入口或基础设施
3. 当前明确包含什么、不包含什么
4. 候选方案与推荐方案
5. 为什么推荐方案最适合当前目标库

确认规则：

- 每次确认只围绕当前收敛结果，不要重新抛出并行问题
- 如果用户调整边界、落点或方案偏好，回到前面的收敛步骤继续澄清，不要直接写规格
- 只有在用户明确认可当前边界、目标与推荐方案后，才能进入规格写作

## 第五步：按唯一写作入口生成 `code-imitation-spec.md`

只有在前四步已经完成、且用户已明确确认当前边界与推荐方案后，才允许进入写作。写入动作本身只包含以下四件事：

1. 做写前就绪检查：确认参考库本质、目标库上下文、双库差异、推荐方案、模块落点、关键接口/状态约束、MVP、非目标与风险已经稳定且具备可写入条件
2. 复用已有输出目录 `docs/analyze-codebase/YYYY-MM-DD-<topic>/`
3. 按下方 `code-imitation-spec.md` 文档结构与章节要求写入完整规格
4. 确保文档显式沉淀足够支撑后续实现规划继续深化的输入信息，而不是把这些信息散落在叙述里

## `code-imitation-spec.md` 文档结构与章节要求

在进入文档结构要求之前，先明确当前产物契约：

- `code-imitation-spec.md` 必须自包含；后续使用者只能依赖文档中显式写出的决策与约束继续细化，不应依赖隐含结构或当前对话补全。
- 下方信息块合在一起，必须把战略边界、推荐方案、模块落点、关键数据 / 接口 / 状态约束、主流程约束、MVP、非目标、验收标准与风险写成显式信息；如果这些内容仍需读者自行拼接，说明规格不合格。
- `code-imitation-spec.md` 不能只给标题、空表格或“后面再想”；若某项暂时没有内容，必须写出“暂无 + 原因”。
- 如果缺少目标库路径、相关模块、技术栈或集成约束，不能进入写规格动作。

`code-imitation-spec.md` 的标题、章节名与编号可以按任务复杂度调整；可以拆章、合章、改名，但必须显式覆盖下面这些信息块。

### 信息块 1：背景与落地目标
必须写清：
- 参考库是谁、目标库是谁
- 本次要迁移的能力与目标使用场景
- 这份规格最终要支撑什么交付
- 成功标准与明确排除范围

### 信息块 2：参考库可迁移本质摘要
必须写清：
- 技术栈与运行环境
- 关键骨架、关键执行链与关键数据/状态模型
- 真正需要迁移的机制、约束与设计本质
- 明显受参考库上下文限制、不能机械照搬的部分

### 信息块 3：目标库上下文与现实约束
必须写清：
- 项目路径或项目地址
- 当前技术栈、运行时与关键基础设施
- 与本次仿写最相关的已有功能、模块、入口、配置或测试入口
- 影响落地方式的现实约束

### 信息块 4：双库差异与迁移难点
必须写清：
- 技术栈差异
- 架构分层与模块组织差异
- 功能现状与可复用能力差异
- 数据/状态/事件流差异
- 这些差异会如何影响落地方案

### 信息块 5：候选方案与推荐方案
必须写清：
- 至少给出足以支撑决策的候选路线；数量按任务复杂度决定，通常为 `2-3` 种
- 每种方案的核心思路、优点、风险、代价与适用边界
- 明确推荐方案，以及放弃其他方案的原因

### 信息块 6：目标库集成策略与模块落点
必须写清：
- 本次能力准备接入目标库的哪些位置
- 准备复用哪些现有能力
- 哪些模块需要扩展，哪些模块需要新增
- 哪些相邻模块、配置、测试或基础设施也必须纳入考虑
- 哪些现有行为不能破坏

### 信息块 7：关键数据、接口与状态约束
必须写清：
- 关键数据结构、状态模型、事件或配置约束
- 核心接口的输入、输出、错误语义与兼容要求
- 主流程上的关键不变量
- 对后续实现最重要的边界条件或限制

### 信息块 8：MVP 范围、非目标与验收
必须写清：
- 第一版必须实现什么
- 明确不做什么
- 成功验收标准是什么
- 哪些测试、观测或人工验证要求必须满足

### 信息块 9：风险、依赖与待确认事项
必须写清：
- 风险
- 依赖
- 待确认事项
- 这些不确定性若不消除，会如何影响后续实现规划或编码执行

### 信息块 10：可直接继续细化的显式决策摘要
必须写清：
- 项目路径
- 目标模块、入口与建议落点
- 推荐方案与禁止偏离的核心约束
- 关键数据结构、接口、状态与主流程约束
- MVP、非目标、测试与验收约束
- 风险、依赖、待确认事项与禁止事项

## 自审要求

reviewer 必须重点检查：

- 是否同时掌握参考库与目标库
- 是否存在目标库信息缺失
- 是否显式写出双库差异、目标库集成策略、模块落点、关键接口/状态约束、MVP、非目标、风险与待确认事项
- 是否已经足以支撑后续实现规划在此基础上继续深化，而不是重新收集战略边界
- 是否仍停留在教学总结，而没有形成规格定义

## Red Flags

- 没有先收集目标库信息就开始写规格
- 只总结参考库，不定义目标库落点
- 把候选方案讨论留空，或把关键决策丢给后续使用者
- 没有定义必须保留什么、可以简化什么、明确不做什么
- 没有写清关键数据/接口/状态约束
- 明知存在高影响歧义，却继续硬写规格
