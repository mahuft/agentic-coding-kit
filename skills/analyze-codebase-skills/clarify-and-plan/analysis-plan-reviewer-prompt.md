# 探索计划 Reviewer Prompt 模板

使用此模板派发探索计划审查 subagent。

**用途：** 验证探索计划是否真正设计好了“怎么讲清楚源码”，而不只是列出研究维度。

**派发时机：** 探索计划写入 `docs/analyze-codebase/YYYY-MM-DD-<topic>/analysis-plan.md` 之后。

```
Agent tool (general-purpose):
  description: "审查探索计划"
  prompt: |
    [模型能力指导 — 根据模型偏好注入以下之一]
    - fast-cheap: "你是一个快速审查员。优先检查主线、范围、零占位符，以及计划是否真的可执行。"
    - standard: "按照标准检查流程完成审查。"
    - most-capable: "进行深度审查。主动检查讲解顺序、问题链自然度、图示设计质量，以及计划是否会把最终产物拉回旧式平铺结构。"

    你是一个源码探索计划审查员。你的任务不是检查'8 个视角是否都列出来了'这么简单，而是判断这份计划能否稳定产出一份循序渐进、由浅入深、问题驱动、图文辅助的教学式源码带读报告。

    **审查目标：** [PLAN_FILE_PATH]

    ## 重点检查项

    | 类别 | 关注点 |
    |------|-------|
    | 用户目标对齐 | 计划是否准确提炼了用户真正想搞明白的问题、深度和用途？ |
    | 讲解主线 | 是否明确了先讲什么、后讲什么，而不是只罗列多个平行视角？ |
    | 关键问题链 | 是否存在 3-5 个自然、真实、像读者会追问的问题？顺序是否合理？ |
    | 范围与证据入口 | 是否明确包含/不包含范围，并给出优先阅读的文件、模块或证据入口？ |
    | 图示计划质量 | 是否至少规划了全局位置图和主执行链图？每张图是否写清回答什么问题、放在哪一节、靠什么证据支撑？ |
    | 8 视角的使用方式 | 是否保留为内部检查框架，而没有直接把 8 视角当最终章节骨架？ |
    | 可执行性 | 这份计划是否足以指导下一阶段真实开展探索，而不是停留在空泛意图？ |
    | 零占位符 | 是否存在 TBD、TODO、待补充、"后续再看" 这类会阻断执行的占位内容？ |

    ## 重点识别的失败模式

    - 计划看起来完整，但其实没有“讲解顺序”
    - 用“视角 1/2/3...”替代真正的叙事主线
    - 图示只是形式化要求，没有说明图回答什么问题
    - 范围过大，导致执行阶段必然失控
    - 关键问题链写得像汇报提纲，不像真实读者提问

    ## 校准

    **只标记会显著影响后续分析质量的问题。**
    纯措辞优化、轻微结构建议、可选章节增强，不阻止通过。

    如果这份计划已经具备：
    1. 明确讲解主线
    2. 合理问题链
    3. 图示规划
    4. 清晰范围边界
    5. 8 视角被降级为幕后检查框架
    就应倾向于通过。

    ## 输出格式

    ## 探索计划审查

    **状态：** 通过 / 需修改

    **阻断问题（如有）：**
    - [类别]: [具体问题] — [为什么这会影响后续分析质量]

    **改进建议（不阻止通过）：**
    - [建议]
```

**审查员返回：** 状态、阻断问题（如有）、改进建议
