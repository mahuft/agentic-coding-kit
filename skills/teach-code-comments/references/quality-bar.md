# 教学式代码注释质量标尺

这份标尺是本 skill 内置的 canonical quality bar。目标不是复制某个外部项目的措辞，而是复制高质量教学式注释应有的分析深度、组织方式和对新手的友好程度。配套示例见 `references/canonical-example.md`。

## 目标效果

读者即使完全不知道仓库背景，也能从注释里逐步建立下面这几个认知层次：

1. 这份文件在系统里扮演什么角色。
2. 每段代码在整条调用链里处于什么位置。
3. 每个关键字段、类型、分支、约定分别由谁生产、谁消费、谁拥有。
4. 哪些行为是框架保证的，哪些是业务层自行处理的。
5. 哪些地方最容易被误读，以及误读后会得到什么错误心智模型。

## 高质量教学式注释的核心特征

### 1. 先交代“为什么存在”，再交代“它是什么”

高质量注释不会只说：

- “这里定义了一个接口”
- “这里调用了一个函数”

而会继续说明：

- 这个接口为什么要存在
- 它在内部协议里承担什么角色
- 它和上游/下游文件如何连接
- 为什么实现上故意保持宽松、严格、可选、动态或惰性

### 2. 注释跟着认知边界走，不跟着语法边界走

高质量教学式注释往往以“一个小知识点”为单位：

- 一组 imports 的职责差异
- 一个上下文对象的字段所有权
- 一个 wrapper 注入的统一能力
- 一个早返回分支背后的约定

因此，切段标准应该是“读者需要在这里建立一个新概念了吗”，而不是“这里正好换行了”。

### 3. 显式补足跨文件信息

这类注释的价值不只在本文件内部，而在于它会把外部上下文压缩回当前注释里。例如：

- 某个类型是给哪个模块消费的
- 某个字段为什么不能由当前层填写
- 某个 helper 最终会把输出写到哪里
- 某个错误为什么继续上抛而不是在此处吞掉

如果不分析邻近模块，这种注释质量做不出来。

### 4. 经常解释“刻意不做什么”

高质量教学式注释会特别指出：

- 为什么这里没有 `try/catch`
- 为什么这里不允许当前层自己填某些字段
- 为什么这里不直接处理截断、持久化、权限、状态写回
- 为什么某个分支故意提前返回

这能帮新手避免把“没写”误读成“忘了写”。

### 5. 特别关注隐藏约定

优先识别这些高价值信息：

- 约定字段或哨兵值，例如某个布尔/可选字段被当作协议信号
- 字段所有权，例如哪些 ID 必须由框架层补齐
- 生命周期或时序依赖，例如初始化阶段和执行阶段拿到的上下文不同
- 统一壳/统一入口，例如 wrapper 在框架层偷偷补上了参数校验、截断、权限等横切能力

### 6. 面向新手，主动指出易错理解

当代码表面语义和真实职责不一致时，要主动提醒：

- “看起来像普通字典，其实是故意保持宽松的协议扩展口”
- “看起来像本地校验，其实错误消息是写给模型重试输入的”
- “看起来像普通字符串，其实对应一次 tool call 的唯一 ID”

## 动笔前分析清单

在插入注释前，优先完成下面这些分析：

1. 目标文件的层级定位：协议层、适配层、业务层、UI 层、持久化层、工具层？
2. 目标文件的主要入口和出口分别是什么？
3. 这段代码依赖哪些外部模块？这些模块提供能力、类型、状态还是约束？
4. 这段代码的输出最终会落到哪里、被谁读取、以什么形式继续传播？
5. 这段代码有哪些副作用：文件读写、网络、数据库、日志、权限、缓存、状态变更？
6. 这里有哪些隐含不变量：命名约定、字段约定、顺序约定、哨兵约定、默认值约定？
7. 这里有哪些“故意缺席”的处理：不 catch、不落盘、不补字段、不做权限、不做重试？
8. 如果删掉这一段，上游或下游会失去什么保障？

如果第 3、4、6、7 项不清楚，说明分析还不够深，先不要写注释。

## 每段注释的推荐骨架

不是每段都要把模板写满，但复杂代码尽量覆盖这些句型：

```ts
// 这段先定义/包装/校验/回写什么，它在当前文件里的职责是什么。
// 这里为什么必须放在这一层，而不是交给上游或下游处理。
// 上游输入来自谁，下游结果会被谁消费，必要时补一句跨文件上下文。
// 这里依赖了什么隐含约定、哨兵值、字段所有权或时序关系。
// 新手最容易误读什么；如果误读，会把它错看成什么。
```

对于简单代码，可以保留其中最有价值的 2-3 句；对于复杂 wrapper、协议、状态流、错误处理代码，通常需要更完整的骨架。

## 哪些位置要特别详细

优先把精力用在这些地方：

- import 段：解释类型导入和运行时导入的职责差异
- 接口、schema、namespace、类型工具
- wrapper、adapter、middleware、hook
- 校验、权限、取消、缓存、截断、落盘
- 错误传播与提前返回
- 通过 `metadata`、状态对象、字符串字面量传递协议语义的地方

## 反例

下面这些都不够格：

- “定义一个变量。”
- “调用这个函数获取结果。”
- “如果条件成立则返回。”
- “这是一个接口，包含若干字段。”

这些句子只复述语法，没有提供新知识。

## 通过标准

交付前问自己：

- 新手只看注释，能否知道这段代码解决什么问题？
- 新手能否知道这段代码和哪些相邻模块协作？
- 新手能否知道这里有哪些隐藏约定和非直观点？
- 删掉这些注释后，读者会不会失去重要背景，而不只是失去“翻译”？

如果答案多数是否定的，就继续分析和重写。
