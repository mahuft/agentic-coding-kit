# Analyze Codebase Skills 优化需求规格说明书

## 1. 文档信息

- 文档日期：2026-04-08
- 适用范围：`analyze-codebase-skills` 工作流中的技能文件与 reviewer prompt
- 目标产物：后续代码改造所依据的统一需求规格

## 2. 背景与问题陈述

`analyze-codebase-skills` 已经具备清晰的阶段化工作流，但当前文本仍存在以下系统性问题：

1. 入口技能 `analyze-codebase` 的 description 依赖“入口”“匹配中文/匹配英文”等硬编码表述，触发语义不够抽象，也不利于迁移到不同平台。
2. `clarify-and-plan` 的 Checklist 把“推断 topic”“创建输出目录”拆成独立任务，弱化了“生成计划”这一核心动作的完整性。
3. 仿写阶段尚未把“参考库”和“目标库”作为正式、一等公民概念纳入输入契约，导致 `clone-writing-plan.md` 与 `clone-writing-report.md` 容易只基于参考库输出抽象方案，缺少与目标库现状的贴合。
4. 多份 reviewer prompt 虽然强调自包含，但仍存在“参考计划/参考报告只用于核对”的上游依赖语义，且尚未全面覆盖“历史上下文污染”和“悬空指代”检查。
5. 多个技能文件仍存在容易产生误解的历史上下文词汇或悬空表达，例如“新的”“保留”“这里”“这层”“这种结构”“这种风格”等；部分表述默认读者已知当前阶段职责或上游产物结构。

本次优化的目标不是改动工作流主干，而是提升整套 skills 系统的触发描述质量、阶段契约清晰度、仿写输入完整性，以及文档自包含性。

## 3. 设计目标

本次改造必须同时满足以下目标：

1. 所有技能 description 改为“在什么场景下使用、满足什么条件时使用”的条件式描述，而不是列举关键词或语言样例。
2. 各阶段职责边界必须显式、可独立理解，不依赖“默认知道上游做过什么”。
3. 仿写阶段必须显式区分“参考库”和“目标库”，并把目标库调研纳入正式工作流输入。
4. 所有 reviewer prompt 必须按照新的阶段契约同步升级，且检查项能直接发现历史上下文污染、自包含性缺陷、隐含职责依赖和参考库/目标库混淆。
5. 下游阶段只能依赖上游文档中显式写出的字段，不能依赖章节隐含结构、作者脑补信息或当前对话上下文。

## 4. 术语定义

### 4.1 参考库

“参考库”指此前由 `clarify-and-plan` 与 `execute-analysis` 负责澄清、探索和分析的代码库，是被学习、拆解和提炼模式的来源。

### 4.2 目标库

“目标库”指用户希望将参考库中的设计、能力或实现思路迁移到的本地项目。目标库不是参考库的别名，也不是参考库的某个目录，而是一个需要被独立理解的落地环境。

### 4.3 仿写

“仿写”指基于参考库提炼出值得迁移的结构、流程、约束与策略，再结合目标库现状生成可实施的落地方案；不等同于照抄参考库，也不等同于脱离目标库输出泛化建议。

### 4.4 自包含

“自包含”指单个技能说明、单个阶段产物、单个 reviewer prompt 在脱离当前对话与隐含上游记忆的情况下，仍能被正确执行或审查。

## 6. 总体约束

### 6.1 文案约束

1. 禁止在 description 中使用“匹配中文”“匹配英文”“例如用户说了 XXX”这类硬编码触发词枚举来定义技能适用场景。
2. 优先使用“Use when ...”或其等价中文语义来描述适用条件、阶段前置条件与职责边界。
3. 禁止使用依赖历史版本或未说明基线的词语，例如“新的”“继续保留”“沿用原有方式”，除非同时显式说明“相对于什么基线”“保留的对象是什么”。
4. 出现“保留”“复用”“简化”“替换”等词语时，必须指明对象和比较基线。
5. 禁止使用无先行词的代词或悬空指代，例如“它”“这层”“这里”“这种结构”“这种风格”“这部分”；如必须使用，需在同段或前一段显式给出先行词。

### 6.2 阶段契约约束

1. 每个技能文件都必须显式说明输入、允许动作、禁止动作、输出、交接规则。
2. 每个阶段都必须说明“下游只能依赖哪些显式字段”，而不是只说“参考上游文档”。
3. 任何阶段都不得假设调用者已经知道工作流结构、章节命名或隐含职责。

### 6.3 reviewer prompt 约束

1. reviewer prompt 只能把参考文件作为一致性校验来源，不能允许 reviewer 用参考文件补齐当前审查对象缺失的信息。
2. reviewer prompt 必须显式检查历史上下文污染、自包含性、悬空指代、阶段职责越界、占位符残留。
3. 与仿写相关的 reviewer prompt 必须显式检查“参考库”和“目标库”是否被正确区分、比较并落到方案中。

## 7. 详细需求

### 7.1 `analyze-codebase/SKILL.md` 优化需求

#### 7.1.1 description 改写

`description` 必须改写为条件式说明，满足以下要求：

1. 删除“源码分析工作流的入口。”这类角色标签式描述。
2. 删除“匹配中文”“匹配英文”等语言硬编码。
3. 直接说明：当对话目标是理解、探索、讲解、拆解、学习或系统化分析某个代码库、模块、机制、执行链、设计决策时使用。
4. 明确说明：此技能在任何澄清问题、代码检索、文件阅读、架构判断之前触发。
5. 风格参考 `using-superpowers` 的条件式描述，但内容必须贴合源码分析工作流自身职责。

#### 7.1.2 主体内容优化

1. 保留“先走工作流、不能直接随意读代码”的强约束。
2. 明确“入口技能只负责识别并转交给 `clarify-and-plan`，不承担计划或分析职责”。
3. 如果保留触发决策流程图，图中的判断条件必须是“场景与意图”，不能退回到关键词匹配。
4. 入口技能不得再出现对语言种类、措辞样例的依赖性说明。

#### 7.1.3 验收标准

满足以下条件视为通过：

1. 仅阅读 `description`，即可知道该技能在什么场景下触发以及触发时机。
2. 删除全部“匹配中文/匹配英文”字样。
3. 删除“源码分析工作流的入口”字样。
4. 任何读者无需知道历史版本内容，也能理解技能职责。

### 7.2 `clarify-and-plan/SKILL.md` 优化需求

#### 7.2.1 Checklist 重构

Checklist 必须重构为“围绕生成计划的工作步骤”，满足以下要求：

1. “推断 topic”和“创建输出目录”不得再作为并列、独立的高层任务存在。
2. 两者应并入“生成 `analysis-plan.md`”这一任务的前置子步骤。
3. 技能文本必须明确：在生成计划文件之前，应基于澄清后的问题表述推断 topic，并创建输出目录。
4. topic 的来源必须是“优化后的问题表述”或“提炼后的分析对象”，而不是原始问题的机械截断。

#### 7.2.2 阶段说明优化

1. 在“生成计划”动作之前，必须有显式的“澄清结果收敛”说明，例如用户目标、讲解主线、关键问题链已具备可写入条件。
2. 输出目录创建动作应被定义为计划产出过程的一部分，而不是与计划本身分离的杂务。
3. 若技能中出现“这种风格”“这里的 8 视角”等表述，应改写为具有明确先行词的表达。

#### 7.2.3 验收标准

满足以下条件视为通过：

1. Checklist 中不再存在独立的“推断 topic”“创建输出目录”条目。
2. 技能文本明确写出“先根据优化后的问题推断 topic，并在写计划前创建目录”。
3. `analysis-plan.md` 的生成逻辑读起来是一个完整闭环，而不是分散的零件清单。

### 7.3 参考库/目标库建模与仿写阶段优化需求

本节是本次改造的重点。虽然用户点名了 `execute-analysis` 和 `execute-clone-writing`，但为保证链路闭环，`clone-writing-plan` 也必须同步纳入改造范围。

#### 7.3.1 通用要求

1. 在仿写阶段相关技能中，必须显式定义“参考库”和“目标库”。
2. 必须明确说明两者职责不同：
   - 参考库负责提供被学习与提炼的模式。
   - 目标库负责提供实际落地约束、技术栈、现有模块与集成边界。
3. 任何仿写计划或仿写报告都不得只基于参考库信息生成。

#### 7.3.2 `execute-analysis/SKILL.md` 追加要求

`execute-analysis` 需要补足供仿写阶段消费的“参考库显式字段”，至少包括：

1. 参考库技术栈与运行环境。
2. 参考库中与后续仿写最相关的架构骨架、关键执行链、关键状态模型、设计约束。
3. 哪些机制属于可迁移本质，哪些机制明显受参考库自身上下文约束。

目的：确保后续阶段不用回头脑补“参考库到底是什么栈、哪些约束可迁移”。

#### 7.3.3 `clone-writing-plan/SKILL.md` 必须新增的前置动作

在生成 `clone-writing-plan.md` 之前，技能必须要求获取并验证目标库信息，至少包括：

1. 目标库路径或项目地址。
2. 目标库中与待仿写能力最相关的已有功能、模块或入口。
3. 用户希望在目标库中实现的能力边界。
4. 目标库当前技术栈、运行时、框架或关键基础设施。
5. 任何会影响落地方式的现实约束，例如代码风格、目录结构、已有抽象层、测试约束。

若上述信息不足，技能必须先发起澄清或最小化目标库扫描，而不是直接输出仿写计划。

#### 7.3.4 `clone-writing-plan/SKILL.md` 必须新增的分析动作

1. 进入目标库做最小必要调研。
2. 比较参考库与目标库在以下维度上的差异：
   - 技术栈与语言
   - 架构分层与模块边界
   - 相关功能现状
   - 状态管理方式
   - 外部依赖与集成方式
   - 可直接复用的模式与必须重写的部分
3. 将差异分析结果显式写入 `clone-writing-plan.md`。

#### 7.3.5 `execute-clone-writing/SKILL.md` 必须新增的报告要求

`clone-writing-report.md` 必须基于“参考库模式 + 目标库现实”共同生成，至少新增以下内容要求：

1. 显式描述本次仿写要如何接入目标库现有结构。
2. 显式描述参考库与目标库在技术栈、模块职责、运行时或工程约束上的关键差异。
3. 对每项核心设计选择，说明是“沿用参考库模式”“按目标库现状改写”还是“折中处理”。
4. Vibe Coding 提示词中必须内联目标库上下文，不能假设执行者知道目标库现状。
5. Vibe Coding 提示词必须说明与目标库哪些现有能力对接、替换或扩展。

#### 7.3.6 `clone-writing-plan.md` 模板新增要求

模板至少应新增以下显式章节或等价字段：

1. 参考库摘要
2. 目标库摘要
3. 参考库与目标库差异对照
4. 目标库集成策略
5. 受目标库约束影响的仿写边界

#### 7.3.7 `clone-writing-report.md` 模板新增要求

模板至少应新增以下显式章节或等价字段：

1. 目标库落地策略
2. 与目标库现有模块的衔接关系
3. 参考库模式如何映射到目标库实现
4. 因目标库差异而做出的调整与取舍

#### 7.3.8 验收标准

满足以下条件视为通过：

1. 仿写相关技能文件中都能清晰区分参考库与目标库。
2. 在没有目标库信息的情况下，工作流不会直接生成仿写计划或仿写报告。
3. `clone-writing-plan.md` 与 `clone-writing-report.md` 的模板中都包含面向目标库的显式字段。
4. Vibe Coding 提示词不再只是“参考库复刻说明”，而是带有目标库接入上下文的落地施工说明书。

### 7.4 reviewer prompt 同步优化需求

#### 7.4.1 全部 reviewer prompt 的共性要求

四份 reviewer prompt 都必须同步加入以下检查维度：

1. 是否存在历史上下文依赖，例如默认知道旧版本文案、旧阶段职责、旧目录结构。
2. 是否存在悬空指代或未定义对象。
3. 是否默认读者知道上游文档的章节结构，而不是仅依赖显式字段。
4. 是否存在“参考文件可补齐缺失信息”的隐性纵容。

#### 7.4.2 `analysis-plan-reviewer-prompt.md` 追加要求

1. 检查 topic 与输出目录是否作为“生成计划”的组成部分，而非独立任务。
2. 检查计划是否从优化后的问题表述出发，而不是从原始提问直接跳到模板。

#### 7.4.3 `analysis-report-reviewer-prompt.md` 追加要求

1. 检查报告是否显式沉淀了可供仿写阶段消费的参考库技术栈、设计约束和可迁移本质。
2. 检查报告是否仍依赖参考计划补齐关键背景。

#### 7.4.4 `clone-plan-reviewer-prompt.md` 追加要求

1. 检查是否显式收集并使用了目标库信息。
2. 检查是否存在参考库/目标库角色混淆。
3. 检查是否包含目标库集成策略与双库差异分析。
4. 若缺少目标库路径、相关模块或技术栈信息，应判定为阻断问题。

#### 7.4.5 `clone-report-reviewer-prompt.md` 追加要求

1. 检查 `clone-writing-report.md` 是否真的落到了目标库，而不是泛化仿写。
2. 检查 Vibe Coding 提示词是否内联了目标库上下文。
3. 检查设计差异与取舍是否同时解释参考库来源与目标库适配原因。
4. 若提示词未说明与目标库已有功能的衔接方式，应判定为阻断问题。

#### 7.4.6 验收标准

满足以下条件视为通过：

1. 四份 reviewer prompt 都新增了历史上下文、自包含性和显式字段依赖检查。
2. 两份仿写 reviewer prompt 都能识别目标库信息缺失或双库角色混淆。
3. reviewer prompt 本身不再依赖隐含的上游章节结构。

### 7.5 全局自包含性与历史上下文治理需求

这是贯穿所有技能与 reviewer prompt 的横切要求。

#### 7.5.1 历史上下文相关要求

1. 禁止默认读者知道“旧内容”“上一版写法”“之前已经约定过什么”。
2. 出现“新增”“继续”“保留”“恢复”“沿用”等词语时，必须说明比较基线或对象。
3. 如果某个阶段需要继承上游结果，必须明确列出继承的字段，而不是用“保留上游结论”概括。

#### 7.5.2 内容自包含性要求

1. 禁止出现没有先行词的“它 / 这层 / 这里 / 这种结构 / 这种风格 / 这部分”。
2. 禁止默认知道工作流阶段职责；每个技能文件都要自解释本阶段边界。
3. 禁止 reviewer prompt 依赖上游产物隐含结构。
4. 禁止使用“见上文”“参考前文”“根据前面内容”补足关键说明。

#### 7.5.3 验收标准

抽样阅读任意单个技能文件或任意单个 reviewer prompt，若脱离当前对话仍能独立理解其职责、输入与审查标准，则视为满足要求。

## 8. 非目标

本次规格不要求：

1. 修改工作流阶段顺序。
2. 改写 8 视角框架本身。
3. 直接产出最终改造后的技能文案。
4. 修改非 `analyze-codebase-skills` 目录下的其他技能系统。

## 9. 实施建议顺序

建议按以下顺序落地改造：

1. 先统一术语与全局写作约束。
2. 再改 `analyze-codebase` description 与入口契约。
3. 再改 `clarify-and-plan` 的 Checklist 和计划生成闭环。
4. 再补 `execute-analysis` 的参考库显式字段。
5. 再补 `clone-writing-plan` 的目标库采集、扫描、差异分析与模板字段。
6. 最后改 `execute-clone-writing` 与四份 reviewer prompt。

原因：仿写阶段的规范依赖前面阶段先把“参考库是什么”与“目标库需要什么”写清楚。

## 10. 交付完成判定

当后续实现满足以下条件时，可视为本规格完成：

1. 九个必改文件均已根据本规格更新。
2. 任意一个技能 description 都以条件和场景定义触发，不再依赖关键词枚举。
3. `clarify-and-plan` 的 Checklist 不再把 topic 推断和目录创建作为独立任务。
4. 仿写阶段明确建模参考库与目标库，并把目标库调研纳入正式输入。
5. 四份 reviewer prompt 都能检查历史上下文污染、自包含性问题和双库职责边界。
6. 随机抽读单个文件时，不需要依赖当前对话或隐含上游结构也能理解其职责与要求。
