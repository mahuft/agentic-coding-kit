# Analyze Codebase Skills V2 优化需求规格说明书

## 1. Summary

- 本规格用于整版替换现有 `docs/analyze-codebase-skills/spec/2026-04-08-skills-optimization-requirements.md`，作为 Analyze Codebase Skills V2 的唯一实现依据。
- 本次改造目标是把后两步从“教学式仿写报告”升级为“面向目标库落地的规格 + 编码任务计划”双阶段流程，解决当前 `clone-writing-report.md` 过度教学化、重复冗余、对 AI 编码约束弱、输出代码质量差的问题。
- 第 1–3 步保留现有分析主干；第 4–5 步进行全链路同步重构：同步修改 skill 名、阶段名、产物名、README、reviewer prompt、上下游消费契约。
- 全流程新增总原则：第 2、3、4、5 步都应尽可能在规划阶段主动暴露不确定性并提问，优先在执行前消灭歧义，而不是执行后补救。

## 2. Public Interfaces

### 2.1 工作流阶段重命名

- 第 4 步：`code-imitation-spec`
- 第 5 步：`coding-task-plan`

### 2.2 产物文件重命名

- `clone-writing-plan.md` → `code-imitation-spec.md`
- `clone-writing-report.md` → `coding-task-plan.md`

### 2.3 reviewer prompt 重命名

- `clone-plan-reviewer-prompt.md` → `code-imitation-spec-reviewer-prompt.md`
- `clone-report-reviewer-prompt.md` → `coding-task-plan-reviewer-prompt.md`

### 2.4 一致性要求

- README、流程图、交付物表、阶段说明、下游引用、文中术语必须全部切换到新命名。
- 本次迁移按原子迁移处理，不保留旧命名兼容层。

### 2.5 上下游显式消费字段

- `execute-analysis` → `code-imitation-spec`：参考库技术栈、运行环境、关键骨架、关键执行链、关键数据/状态模型、可迁移本质、上下文限制。
- `code-imitation-spec` → `coding-task-plan`：目标库路径、目标库技术栈、相关现有模块/入口、目标能力边界、双库差异、集成策略、必须保留骨架、允许简化项、明确不做项、验收标准、待确认事项。

## 3. Implementation Changes

### 3.1 全局提问原则（适用于第 2–5 步）

以下原则必须显式写入 `clarify-and-plan`、`execute-analysis`、`code-imitation-spec`、`coding-task-plan`：

1. 能在规划或产出前确认的信息，不要拖到执行中再发现。
2. 遇到影响方向、边界、接口、集成方式、验收标准的高影响不确定性时，必须优先提问。
3. 提问应一次一个、聚焦决策，优先消除会导致后续返工的问题。
4. 若可通过轻量扫描/检索确认事实，先探索；若属于产品意图或取舍，再问用户。
5. 第 3 步虽然是执行分析，但仍应在真正深挖前暴露关键缺口；若 `analysis-plan.md` 无法支撑可靠分析，应先回到澄清，而不是硬做。

### 3.2 README 与总流程

- 将第 4 步定义为“代码仿写需求规格说明书”，职责是承上启下：把参考库分析结果与目标库现实约束合并成可审查的规格。
- 将第 5 步定义为“编码任务计划说明书”，职责是把第 4 步规格转成可直接发送给新的 Coding Agent 的超详细 Agentic Coding 提示词。
- README 中删除“第 5 步产出 5 个交付物”的表述；新定义为“第 5 步仅产出 1 个 `coding-task-plan.md`，整份文档即提示词”。

### 3.3 第 4 步：`code-imitation-spec`

#### 3.3.1 阶段职责

- 必须显式声明双库模型：
  - 参考库：被学习与提炼模式的来源；
  - 目标库：能力真正落地的项目。
- 该阶段负责把参考库分析结果与目标库现实约束转成可审查规格，而不是直接生成编码提示词。

#### 3.3.2 执行方式

必须吸收 `superpowers/brainstorming` 的方法，固定为：

1. 读取 `analysis-report.md`
2. 对目标库做最小必要扫描
3. 一次一个问题地向用户澄清目标库信息与成功标准
4. 提出 2–3 种可行落地方案，说明 trade-off，并给出推荐方案
5. 形成 `code-imitation-spec.md`
6. reviewer 自审
7. 用户批准后才能进入第 5 步

若 reviewer 发现阻断缺口，允许返回澄清问题，而不是硬写规格。

#### 3.3.3 `code-imitation-spec.md` 必备内容

- 背景与目标
- 参考库摘要
- 目标库摘要
- 双库差异对照
- 目标库集成策略
- 候选方案与推荐方案
- MVP 范围/非目标
- 必须保留的本质
- 允许简化与后置复杂性
- 风险、依赖、待确认事项
- 进入第 5 步所需的显式输入字段

#### 3.3.4 明确禁止

- 在目标库信息缺失时直接产出规格
- 把参考库分析结论原样平移为目标库方案
- 提前生成 `coding-task-plan.md` 正文

### 3.4 第 5 步：`coding-task-plan`

#### 3.4.1 阶段职责

- 唯一产物为 `coding-task-plan.md`，整份文档就是给新的 Coding Agent 的执行提示词。
- 不再保留“Vibe Coding 提示词”作为内部子章节概念。
- 文档必须直接内联当前项目的具体上下文，不能使用“目标库”“参考库”这类工作流内部标签。

#### 3.4.2 自包含要求

- 允许内联从来源实现提炼出的必要事实。
- 禁止出现“见参考库 / 见 analysis-report / 参考前文”。
- 禁止要求新的 Coding Agent 依赖来源上下文。

#### 3.4.3 生成过程要求

生成过程也要吸收 `superpowers/brainstorming` 方法：

1. 在写某章前若发现方向、接口、边界、优先级、验收口径仍不够明确，必须主动提问。
2. 若存在更优实现路线或更稳妥的落地方案，可以在生成过程中提出方案建议与推荐，再继续写章节。
3. 目标是把不确定性尽量前移到章节生成前，而不是写完整文后再暴露问题。

#### 3.4.4 文风要求

内容风格从“教学解释”切换为“执行说明”，必须减少重复背景、减少说教、减少平行罗列，强化模块职责、数据契约、流程约束、实施顺序、测试验收；同时允许文档足够长，以保证新的 Coding Agent 能稳定生成高质量代码。

#### 3.4.5 章节门控

`coding-task-plan.md` 采用单文件分章门控，章节结构至少覆盖以下信息块，并可随复杂度扩展：

1. 任务定义与项目上下文
2. 范围、非目标与成功标准
3. 现有代码上下文与复用策略
4. 目标设计与模块职责
5. 数据结构、接口契约与状态模型
6. 主流程、边界流程与伪代码
7. 实施顺序、测试与验收

生成规则：

- 每完成一章即写入/更新同一个 `coding-task-plan.md`
- 每章完成后必须请求用户批准
- 未获批准不得继续下一章
- 若章节前发现重大不确定性，先提问或提方案，不得硬写
- 最后一章通过后再做一次全文 reviewer 自审
- 最终文档不得继续使用“目标库”“参考库”这类工作流内部术语，而必须直接写出具体项目路径、模块名、文件路径、接口名与约束

### 3.5 Reviewer Prompts

#### 3.5.1 `code-imitation-spec-reviewer-prompt.md`

必须重点检查：

- 是否同时掌握参考库与目标库
- 是否存在目标库信息缺失
- 是否给出目标库集成策略、MVP、非目标、风险与进入第 5 步所需显式字段
- 是否仍停留在教学总结而非规格定义

#### 3.5.2 `coding-task-plan-reviewer-prompt.md`

必须支持两级审查：

- 章节级审查：当前章节是否足够驱动实现
- 全文级审查：整份提示词是否自包含、是否完全服务当前项目、是否无外部依赖

两份 reviewer prompt 都必须检查：

- 历史上下文污染
- 悬空指代
- 参考库/目标库角色混淆
- 占位符、空表格、缺失约束
- 是否存在“教学味过重、执行约束不足”的失败模式

`coding-task-plan-reviewer-prompt.md` 还要检查：

- 当前章节或全文是否仍有应当在写作前先澄清的问题未被前置处理
- 是否存在更优方案被忽略且会显著影响后续编码质量
- 是否仍然使用“目标库”“参考库”这类工作流元术语而不写具体项目上下文

## 4. Test Plan

### 4.1 流程一致性

- README 流程图、阶段名称、交付物表、skill 说明、reviewer prompt 命名完全一致，无旧术语残留。

### 4.2 全局提问原则

- 第 2–5 步的技能文档中都能明确读出“尽可能前置澄清”的统一原则。
- 对于高影响缺口，skill 行为应是先澄清/回退，而不是继续硬写。

### 4.3 第 4 步场景

- 仅提供 `analysis-report.md`、未提供目标库路径时，skill 必须阻塞并发起澄清。
- 目标库信息齐全时，skill 必须先提问/扫描，再生成规格。
- reviewer 发现目标库模块、约束或成功标准缺失时，结论必须是阻断。

### 4.4 第 5 步场景

- 输入一个合格的 `code-imitation-spec.md` 后，输出必须是单一 `coding-task-plan.md`，而不是报告 + 提示词混合包。
- 任意章节单独阅读时，必须能看出它在当前项目中要实现什么。
- 整文不得出现“见参考报告 / 见上文 / 参考库里……”这类外部依赖句式。
- 整文不得出现“目标库”“参考库”这类对新的 Coding Agent 无意义的工作流元术语。
- 每章结束后必须停下来等待批准；未批准不能继续生成后续章节。
- 若章节生成前仍存在影响实现的关键歧义，skill 必须先提问或提出更优方案，而不是直接继续写。

### 4.5 质量回归

- 用现有 `clone-writing-report.md` 示例做对照审查时，新规格应能明确判定其问题：教学化、重复、引用参考库过多、目标库上下文不足、执行约束不够。

## 5. Assumptions

- 输出目录仍沿用 `docs/analyze-codebase/YYYY-MM-DD-<topic>/`，本次只替换第 4/5 步的文件名与内容契约。
- 第 1–3 步的核心能力不重做，仅补充对第 4 步有用的参考库显式字段与前置澄清原则。
- 本次规格是破坏性命名迁移，实现时应一次性同步修改所有内部引用，不保留旧 skill 名作为正式接口。
- 旧规格文件可保留作历史记录，但新实现必须以 V2 规格为唯一准绳。
