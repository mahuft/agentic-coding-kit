# 执行分析计划，产出“教学式源码带读报告”

你的任务不是把 8 个视角分别写成 8 个孤立章节，而是基于已批准的计划，把源码分析写成一份真正帮助工程师理解代码的带读文档：先建立地图，再走主链，再拆机制，再讲设计动机与坑点。

<HARD-GATE>
- 必须优先使用检索与精准阅读，避免无序通读
- 8 视角只能作为内部检查框架，不能直接成为最终报告主结构
- 至少包含 2 张 Mermaid 图：全局位置图 + 主执行链图
- 图必须在正文中被解释，不能只是附录
</HARD-GATE>

## 全局提问原则

本工作流要求：**尽可能在规划阶段提出问题，好过于真正执行后再发现问题。**

在本阶段必须遵守以下原则：

1. 开始深挖前，先检查 `analysis-plan.md` 是否足以支撑可靠分析。
2. 若缺少会影响分析方向、主执行链选择、关键问题拆解或图示计划的高影响信息，必须先回到澄清，而不是硬做。
3. 若可通过高层检索或精确定位补齐事实，先探索；若属于用户意图、边界判断或分析优先级，再问用户。
4. 在真正深挖前暴露关键缺口，好过于写完整份报告后才发现主线跑偏。

## 术语约定与阶段契约

- **分析对象**：指已在 `analysis-plan.md` 中被明确的目标系统、模块、机制、执行链或设计问题。后文所有“地图 / 主链 / 机制 / WHY / 坑点”都围绕分析对象展开。
- **参考库**：指当前被分析和学习的代码库。后续阶段会把这份分析报告当作学习来源，因此本阶段必须显式沉淀参考库信息，而不是只在脑中理解。
- **当前阶段产物**：指本阶段唯一要生成的 `analysis-report.md`。
- **输入**：一份已获用户批准的 `analysis-plan.md`，其分析对象位于参考库中。
- **允许动作**：基于计划做高层检索、精确定位、阅读关键代码片段，补齐主线环境与相邻环境，扩展关键问题与证据，生成并验证 Mermaid 图，撰写分析报告，派发 reviewer 自审。
- **禁止动作**：跳过计划直接分析、依赖隐含上下文补完计划缺失字段、把 8 视角直接平铺为最终报告章节、把报告写成对计划的轻量改写或摘要复述。
- **输出**：一份可独立阅读、可供后续迁移或实现规划直接使用的 `analysis-report.md`，至少要覆盖结论、地图、主执行链、关键问题拆解、WHY/权衡、坑点与调试抓手，以及自包含的参考库摘要；该报告必须是对 `analysis-plan.md` 的扩写与深化，而不是改写版摘要。
- **交接规则**：后续使用者只能依赖本报告中显式写出的核心骨架、关键机制、设计约束、参考库技术栈、运行环境、关键数据/状态模型、可迁移本质和明显受参考库上下文限制的部分，不应默认知道分析阶段脑中补过的隐含结构。

## 必须遵守的执行顺序

下面的顺序是强约束执行流，不是参考建议。若后文任何展开段落看起来允许跳步，以这里为准。

1. 读取已获用户批准的 `analysis-plan.md`
2. 只提取并消费计划中显式写出的下列字段：
   - 当前问题表述与学习目标
   - 已确认的关键澄清结论
   - 一句话定位
   - 候选探索方法与推荐方法
   - 讲解主线
   - 关键问题链
   - 图示计划
   - 8 视角检查表
   - 范围边界、证据入口、避坑与调试线索
3. 先做计划完备性检查：
   - 若缺少会影响主线、主执行链、关键问题拆解、图示或范围判断的阻断性前提，必须回退到 `clarify-and-plan` 补齐
   - 只有非阻断性遗漏，才允许在后续研究中补全，并在报告中显式写出“计划基线之外补充了什么”
4. 按计划中的推荐探索方法组织第一轮检索顺序；若计划未明确推荐方法，才退回按讲解主线组织
5. 先补齐主线环境与相邻环境：
   - 主线环境：主轴相关入口、核心模块、关键对象、关键状态、主执行链上下游
   - 相邻环境：会影响理解主线的配置、测试、扩展点、边界模块、错误处理或协作模块
6. 以计划中的关键问题链为基线，扩展出更细的分析问题与分组
   - 必须优先补充计划中尚未展开但执行阶段证明重要的关键问题
   - 禁止脱离计划另起一套平行主线
7. 围绕扩展后的问题链补齐代码证据、图示证据、设计权衡证据与调试线索
8. 用 8 视角做查漏补缺，只吸收入正文需要的补充信息，不要把 8 视角平铺成报告章节
9. 梳理参考库技术栈、运行环境、关键骨架、关键执行链、关键数据/状态模型、可迁移本质和上下文约束
10. 按“计划基线 -> 本轮补充环境 -> 深化问题拆解 -> WHY/权衡 -> 坑点/调试 -> 参考库摘要”的叙事顺序生成并验证图示
11. 按下方文档契约写出 `analysis-report.md`
12. 派发 reviewer 自审
13. 等待用户批准

## 分析目标

最终报告应让读者做到：

1. 用一句话说清这个模块/系统是干嘛的
2. 说清当前分析对象在整体架构中的位置
3. 讲出一条核心执行链是如何跑完的
4. 理解几个最关键的对象/状态/协作点
5. 知道为什么作者这样设计，而不是另一种写法
6. 知道哪些地方最容易误解、排障从哪里入手
7. 为后续迁移或实现规划提供足够明确的参考库上下文
8. 即使此前从未接触过这个代码库，也能建立起可靠的心智模型

## 写作原则：把读者当作第一次接触这个库的工程师

- **默认读者零背景**：不要假设读者知道模块名、缩写、目录职责、运行时约束或常见术语；第一次出现就解释清楚。
- **简单知识点也要讲**：如果跳过某个“看起来 obvious”的概念会让读者断线，就必须补上。
- **先搭脚手架，再进细节**：先告诉读者“这是什么 / 在哪里 / 为什么重要”，再进入函数、状态和代码路径。
- **事实与判断分开写**：代码能直接证明的写成事实；基于证据的推断要明确说明这是你的判断。
- **为新手补认知桥梁**：必要时加“范围声明 / 前置概念 / 术语解释 / 如何阅读本报告”这类导读，不要怕啰嗦。
- **不要被模板反向驱动**：问题拆解、图示、分节数量都应服从主题复杂度和教学效果，而不是服从固定小节数。

## 基于计划的深化研究方式

### 先把计划当成研究基线
- 不要从零决定讲什么；先锁定计划已经确认的主线、问题链、图示与范围边界
- 报告的目标不是“把计划改写成完整句子”，而是在计划骨架上补充更完整的环境、证据与解释

### 先补环境，再深挖主链
- 先补齐主线环境：分析对象属于哪个子系统、与哪些模块协作、主链入口在哪里、关键对象和状态从何而来
- 再补齐相邻环境：哪些配置、测试、扩展点、错误处理、边界模块或上下游关系会影响主线理解
- 只有环境足够稳定后，才开始沿主执行链做精读

### 再把计划中的问题链扩成完整研究问题集
- 计划里的关键问题链是起点，不是上限；执行阶段要把真正重要但尚未展开的问题继续细化、分组和排序
- 问题扩展应优先服务于三类增量：主线环境遗漏、关键机制未解释透、调试/修改时最容易踩坑的点
- 每个问题都必须有代码证据；新增问题也必须能回挂到计划主线，而不是另起主题

### 最后把计划中的避坑线索落成可操作结论
- 解释设计动机、替代方案与取舍
- 把计划里的“预计坑点/调试线索”落成真实可用的排障入口、观察现象、优先检查点与修改风险
- 显式标出哪些结论可迁移到其他项目，哪些结论受参考库自身上下文限制

## 工具策略

### 第一层：高层检索
- `codebase-retrieval`：找相关模块、调用方、下游依赖、测试

### 第二层：精确定位
- 搜索符号、函数名、类型名、关键状态字段

### 第三层：精读关键片段
- 只读与主线相关的代码片段
- 避免把 token 浪费在与当前问题无关的大段实现上

## 写作要求：必须像在带一个工程师读代码

### 强制使用“先总后分、先搭地图再进实现”的讲解弧线

在写报告之前，先明确报告契约：

- `analysis-report.md` 必须能在不查看 `analysis-plan.md` 的情况下被单独理解；参考计划只用于校验一致性，不用于补全缺失上下文。
- `analysis-report.md` 必须明确体现“计划基线”与“本轮扩展点”：读者应该能看出报告在计划基础上补齐了哪些环境、问题、证据和调试线索。
- **全局位置图** 与 **主执行链图** 至少各 `1` 张，且图后必须解释“这张图怎么读”与“这张图证明了什么”；复杂主题应主动增加更多图。
- **关键问题拆解** 的数量必须服从主题复杂度，至少要覆盖主线中的主要认知断点与执行阶段新增发现；每个问题都必须包含“短答案 / 代码证据 / 展开解释 / 一句话总结”。
- **代码证据** 至少应包含文件路径与符号名；必要时补充行号。不要只写“见某文件”这类弱证据。
- **参考库摘要** 必须显式写出技术栈、运行环境、关键骨架、关键执行链、关键数据/状态模型、可迁移本质和上下文约束，确保后续迁移或实现规划可以直接使用。
- **避坑与调试内容** 必须比计划更具体：不仅说明“哪里可能有坑”，还要说明如何观察、如何定位、优先检查什么。
- 如果某一栏没有信息，必须写出“暂无 + 原因”，不能留空表格或空 bullet。
- 第一次出现的术语、缩写、模块名、状态名，都要写出“它是什么 / 在哪里 / 为什么重要”，不要默认读者知道。

## `analysis-report.md` 文档结构与章节要求

`analysis-report.md` 不要求机械复刻固定标题；你可以按主题拆分、合并、重命名章节，但整体叙事必须从“快速结论 -> 全局地图 -> 主执行链 -> 关键机制 -> WHY/权衡 -> 坑点/调试 -> 参考库摘要”逐步推进。必须显式覆盖下面这些信息块。

### 信息块 1：开头导读
必须写清：
- 一句话定位
- 若干最重要的结论（数量按复杂度决定，不必硬凑 3 条，也不要被 3 条限制）
- 报告覆盖范围与明确排除范围
- 计划基线与本轮扩展点：本报告沿用了计划中的哪条主线，又补齐了哪些主线环境、相邻环境、关键问题或调试线索
- 这份报告推荐怎么读；必要时补充前置概念或术语说明

### 信息块 2：全局地图与协作边界
必须写清：
- 当前分析对象在系统中的位置
- 上游、下游、边界与关键协作方
- 计划主线之外但会影响理解的相邻环境：例如配置、测试、扩展点、边界模块或错误处理路径
- 至少 1 张全局位置图
- 图的阅读方式，以及这张图帮助读者建立了什么整体认知

### 信息块 3：主执行链
必须写清：
- 选择哪条链路作为主链，以及为什么选它
- 这条链路如何从入口走到输出
- 至少 1 张主执行链图
- 按步骤拆解关键节点，每一步都给出代码证据和解释

### 信息块 4：关键问题拆解
必须写清：
- 围绕读者自然会追问的问题展开，而不是围绕功能清单展开
- 先覆盖计划中的关键问题，再显式补充执行阶段新增的重要问题；必要时说明这些新增问题为什么必须补进来
- 问题数量服从复杂度；复杂主题应继续拆，不要压成几个大标题
- 每个问题都要包含短答案、代码证据、展开解释和一句话总结
- 必要时可把问题分组，例如“运行机制”“状态流转”“权限与边界”“扩展点”等

### 信息块 5：设计动机与权衡
必须写清：
- 关键设计决策
- 每个决策带来的好处、代价、约束与替代方案
- 你基于证据做出的设计判断，以及哪些部分属于推断

### 信息块 6：容易误解的点、调试抓手与修改风险
必须写清：
- 计划中预判的常见误区，以及执行阶段额外发现的误区
- 隐藏陷阱或隐性耦合
- 排障入口、调试抓手、观察现象与优先检查点
- 如果读者后续要改代码，最容易踩坑的地方

### 信息块 7：可直接用于后续迁移或实现规划的参考库摘要
必须写清：
- 技术栈与运行环境
- 关键骨架、关键执行链与关键数据/状态模型
- 可迁移的机制与设计约束
- 明显依赖参考库上下文的部分
- 如果迁移到其他项目，优先注意的适配点

### 信息块 8：8 视角补充检查
必须写清：
- 8 视角各自补充了什么信息
- 哪些信息已经吸收入正文
- 哪些视角只作为幕后校验，不需要单独展开成正文大章

### 信息块 9：延伸探索建议
按需写清：
- 如果读者下一步想继续深挖，最值得读哪些文件、机制或实验路径
- 为什么这些延伸方向重要

## Red Flags

- 直接把 8 视角写成最终章节
- 报告整体只是把 `analysis-plan.md` 换一种措辞重写，没有明显新增环境、证据、问题或调试信息
- 计划存在阻断性缺口，却不回退补计划，而是靠脑补继续写报告
- 全篇都在说“是什么”，几乎不解释“为什么”
- 没有一条完整主执行链
- 明知计划缺少关键前提，仍继续深挖并输出确定性结论
