# 设计仿写计划：把分析报告转成“可执行复刻蓝图”

你的任务不是生成仿写内容，而是先把仿写策略设计清楚：到底要忠实复刻原实现的哪些骨架、砍掉哪些复杂性、提示词怎样做到真正自包含、以及最终的仿写报告该如何帮助另一个 AI 直接产出高质量代码。

<HARD-GATE>
此阶段只做计划，不生成伪代码、不生成 Vibe Coding 提示词正文、不生成术语词典条目。只规划框架、边界、模板和质量标准。
</HARD-GATE>

## 术语约定与阶段契约

- **本次仿写目标范围**：指当前仿写计划准备忠实复刻或有意简化的那部分能力边界，来源于已批准的 `analysis-report.md` 中显式写出的核心骨架、关键机制和设计约束。
- **当前阶段产物**：指本阶段唯一要生成的 `clone-writing-plan.md`。
- **输入**：一份已获用户批准的 `analysis-report.md`。
- **允许动作**：提取分析报告中的显式发现，定义仿写边界、MVP 策略、伪代码提炼规划、术语词典规划、设计差异规划与 Vibe Coding 模板蓝图。
- **禁止动作**：直接生成伪代码正文、直接生成 Vibe Coding 提示词正文、依赖分析阶段的隐含上下文补齐计划缺口。
- **输出**：一份可被 `execute-clone-writing` 显式消费的 `clone-writing-plan.md`，至少要写清复刻边界、简化策略、明确不做项、Vibe Coding 模板蓝图与风险点。
- **交接规则**：执行阶段只应依赖本计划中显式写出的字段，不应默认知道分析报告的章节设计或作者脑中的隐含结构。

## 核心理念

- 仿写不是“复制报告内容”，而是“提炼出可被另一个 AI 稳定执行的最小骨架”。
- 最重要的交付物是 **Vibe Coding 提示词**，它必须是零外部依赖、拿去就能生成高质量代码的完整施工说明书。
- 仿写计划必须先回答：
  1. 复刻什么本质
  2. 放弃什么复杂性
  3. 用什么结构把这些信息交给另一个 AI

## Checklist

1. 读取已获用户批准的 `analysis-report.md`
2. 只提取显式写出的核心骨架、关键机制、设计约束与取舍
3. 定义本次仿写目标范围与最小可行边界
4. 设计 `clone-writing-report.md` 结构
5. 规划伪代码提炼策略
6. 规划术语词典提炼策略
7. 规划设计差异分析策略
8. 规划改进建议策略
9. 重点设计 Vibe Coding 提示词标准模板
10. 生成 `clone-writing-plan.md`
11. 派发 reviewer 自审
12. 等待用户批准

## 第一原则：先定义“仿写什么”，再定义“怎么写提示词”

在计划中必须明确：

- **必须忠实复刻的骨架**：哪些结构、流程、数据流、模块关系是不能丢的
- **允许简化的部分**：哪些是为工程成熟度服务的复杂性，可暂时砍掉
- **禁止偏离的约束**：哪些设计原则必须保住，不然就不叫“仿写”了

## 第二原则：Vibe Coding 提示词必须是“施工图纸”，不是灵感描述

计划中必须把 Vibe Coding 提示词视为核心交付物，并为它设计一套严格模板。

### 这份提示词必须包含

1. 你要构建的系统是什么
2. 为什么存在、解决什么问题
3. 目标技术栈与工程边界
4. 需要实现的模块与职责
5. 一条主执行链如何跑通
6. 关键数据结构/状态模型
7. 必须遵守的设计约束
8. 伪代码骨架
9. 术语词典
10. 验收标准
11. 禁止事项
12. 实施顺序建议
13. 测试/验证要求

## 第三原则：仿写报告也要像教学式施工手册

最终的 `clone-writing-report.md` 不应只是 5 份资料堆在一起，而应该帮助用户和另一个 AI 理解：

- 原实现最值得抄的是什么
- 先抄哪一层最划算
- 哪些复杂性可以后置
- 如何用一个详细提示词把这些要求完整传递出去

## `clone-writing-plan.md` 模板

在进入模板之前，先明确下游消费契约：

- `execute-clone-writing` 只应依赖本计划中显式写出的 **仿写目标、必须保留的骨架、可后置复杂性、明确不做项、伪代码提炼规划、术语词典规划、设计差异规划、Vibe Coding 模板蓝图、风险点**。

## 重点：Vibe Coding 提示词模板设计要求

计划中必须明确，执行阶段生成的 Vibe Coding 提示词要具备以下特点：

### A. 像项目 PRD + 技术设计 + 实施说明书的混合体
不能只是：
- “请帮我实现一个类似 XX 的系统”

而要像：
- 背景
- 目标
- 范围
- 模块职责
- 数据结构
- 关键流程
- 伪代码
- 术语
- 验收标准
- 禁止事项

### B. 明确分层
提示词要把信息分层提供给另一个 AI：

1. 为什么做
2. 做什么
3. 怎么组织
4. 核心流程怎么跑
5. 关键约束有哪些
6. 如何判断完成

### C. 允许详细
提示词不是越短越好，而是越能减少误解越好。只要结构清晰，就可以写得很详细。

## Red Flags

- 把仿写计划写成 5 个交付物的平行表格，缺少主策略
- 没有明确“保留什么本质 / 放弃什么复杂性”
- 把 Vibe Coding 提示词当成普通附属章节
- 计划里没有定义提示词质量标准
- 范围不清，导致执行阶段容易过度实现或跑偏

## 自审要求

reviewer 重点看：
- 仿写计划是否已经把“复刻策略”讲清楚
- 是否把 Vibe Coding 提示词提升为核心交付物
- 是否已设计出真正可执行的提示词模板蓝图
- 是否明确了保留、简化、放弃三类边界

## 下一步
