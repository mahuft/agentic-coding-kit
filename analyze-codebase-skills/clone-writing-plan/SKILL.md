# 设计仿写计划：把分析报告转成“可执行复刻蓝图”

你的任务不是生成仿写内容，而是先把仿写策略设计清楚：到底要从参考库学习什么本质、怎样把这些模式接入目标库、哪些复杂性要后置、以及最终的仿写报告该如何帮助另一个 AI 直接产出高质量方案。

<HARD-GATE>
此阶段只做计划，不生成伪代码正文、不生成 Vibe Coding 提示词正文、不生成术语词典条目。只规划框架、边界、模板和质量标准。
</HARD-GATE>

## 术语约定与阶段契约

- **参考库**：指此前由 `clarify-and-plan` 与 `execute-analysis` 分析和学习的代码库，是本次仿写要学习模式、结构和约束的来源。
- **目标库**：指用户希望落地仿写结果的本地项目。目标库不是参考库的别名，而是一个需要被单独理解其技术栈、现有功能和集成边界的项目。
- **本次仿写目标范围**：指当前仿写计划准备忠实复刻或有意简化的能力边界，必须同时受到参考库发现和目标库现实约束的共同定义。
- **当前阶段产物**：指本阶段唯一要生成的 `clone-writing-plan.md`。
- **输入**：一份已获用户批准的 `analysis-report.md`，以及目标库的路径、相关功能、技术栈与集成约束等显式信息。
- **允许动作**：提取分析报告中的显式发现，澄清目标库信息，进入目标库做最小必要调研，比较参考库与目标库差异，定义仿写边界、MVP 策略、伪代码提炼规划、术语词典规划、设计差异规划与 Vibe Coding 模板蓝图。
- **禁止动作**：在缺少目标库关键信息时直接生成计划、把目标库默认成与参考库相同、直接生成伪代码正文或 Vibe Coding 提示词正文、依赖分析阶段的隐含上下文补齐计划缺口。
- **输出**：一份可被 `execute-clone-writing` 显式消费的 `clone-writing-plan.md`，至少要写清双库背景、差异对照、目标库集成策略、复刻边界、简化策略、明确不做项、Vibe Coding 模板蓝图与风险点。
- **交接规则**：执行阶段只应依赖本计划中显式写出的字段，不应默认知道分析报告的章节设计、作者脑中的隐含结构，或目标库的额外背景。

## 核心理念

- 仿写不是“复制报告内容”，而是“提炼参考库中值得迁移的本质，并把它们变成适合目标库落地的可执行方案”。
- 最重要的交付物是 **Vibe Coding 提示词**，它必须是零外部依赖、拿去就能生成高质量代码的完整施工说明书。
- 仿写计划必须先回答：
  1. 参考库中最值得学习的本质是什么
  2. 目标库现在是什么样
  3. 两个代码库之间有哪些关键差异
  4. 这些差异会怎样影响落地方式

## Checklist

1. 读取已获用户批准的 `analysis-report.md`
2. 只提取显式写出的参考库骨架、关键机制、设计约束、技术栈与上下文限制
3. 收集并验证目标库信息
   - 目标库路径或项目地址
   - 目标库中与待仿写能力最相关的已有功能、模块或入口
   - 用户希望在目标库中实现的能力边界
   - 目标库当前技术栈、运行时、框架或关键基础设施
   - 任何会影响落地方式的现实约束
4. 如果目标库信息不足，先发起澄清；必要时进入目标库做最小必要调研
5. 比较参考库与目标库在技术栈、架构分层、相关功能现状、状态管理、依赖集成与可复用模式上的差异
6. 定义本次仿写目标范围与最小可行边界
7. 设计 `clone-writing-report.md` 结构
8. 规划伪代码提炼策略
9. 规划术语词典提炼策略
10. 规划设计差异分析策略
11. 规划改进建议策略
12. 重点设计 Vibe Coding 提示词标准模板
13. 生成 `clone-writing-plan.md`
14. 派发 reviewer 自审
15. 等待用户批准

## 第一原则：先定义“从参考库学什么、往目标库放哪里”，再定义“怎么写提示词”

在计划中必须明确：

- **参考库中必须忠实复刻的骨架**：哪些结构、流程、数据流、模块关系是不能丢的
- **目标库中的落点**：这些骨架会接到哪些现有模块、入口、状态层或基础设施上
- **允许简化的部分**：哪些是为工程成熟度服务的复杂性，可暂时砍掉
- **禁止偏离的约束**：哪些设计原则必须保住，不然就不叫“仿写”了

## 第二原则：Vibe Coding 提示词必须是“施工图纸”，不是灵感描述

计划中必须把 Vibe Coding 提示词视为核心交付物，并为它设计一套严格模板。

### 这份提示词必须包含

1. 你要构建的系统是什么
2. 为什么存在、解决什么问题
3. 参考库中最值得学习的本质
4. 目标库现状与相关模块
5. 目标技术栈与工程边界
6. 需要实现的模块与职责
7. 一条主执行链如何跑通
8. 关键数据结构/状态模型
9. 必须遵守的设计约束
10. 伪代码骨架
11. 术语词典
12. 验收标准
13. 禁止事项
14. 实施顺序建议
15. 测试/验证要求

## 第三原则：仿写报告也要像教学式施工手册

最终的 `clone-writing-report.md` 不应只是 5 份资料堆在一起，而应该帮助用户和另一个 AI 理解：

- 参考库最值得抄的是什么
- 目标库目前已有些什么
- 先抄哪一层、接到哪里最划算
- 哪些复杂性可以后置
- 如何用一个详细提示词把这些要求完整传递出去

## `clone-writing-plan.md` 模板

在进入模板之前，先明确下游消费契约：

- `execute-clone-writing` 只应依赖本计划中显式写出的 **参考库摘要、目标库摘要、双库差异对照、目标库集成策略、仿写目标、必须保留的骨架、可后置复杂性、明确不做项、伪代码提炼规划、术语词典规划、设计差异规划、Vibe Coding 模板蓝图、风险点**。

## 重点：Vibe Coding 提示词模板设计要求

计划中必须明确，执行阶段生成的 Vibe Coding 提示词要具备以下特点：

### A. 像项目 PRD + 技术设计 + 实施说明书的混合体
不能只是：
- “请帮我实现一个类似 XX 的系统”

而要像：
- 背景
- 双库上下文
- 目标
- 范围
- 模块职责
- 数据结构
- 关键流程
- 伪代码
- 术语
- 验收标准
- 禁止事项

### B. 明确分层
提示词要把信息分层提供给另一个 AI：

1. 为什么做
2. 参考库中最值得学习的本质是什么
3. 目标库现在有什么
4. 要做什么
5. 怎么组织
6. 核心流程怎么跑
7. 关键约束有哪些
8. 如何判断完成

### C. 允许详细
提示词不是越短越好，而是越能减少误解越好。只要结构清晰，就可以写得很详细。

## Red Flags

- 把仿写计划写成 5 个交付物的平行表格，缺少主策略
- 没有明确“保留什么本质 / 放弃什么复杂性”
- 没有先收集目标库信息就开始写计划
- 没有比较参考库和目标库的关键差异
- 把 Vibe Coding 提示词当成普通附属章节
- 计划里没有定义提示词质量标准
- 范围不清，导致执行阶段容易过度实现或跑偏

## 自审要求

reviewer 重点看：
- 仿写计划是否已经把“复刻策略”讲清楚
- 是否把目标库信息和集成策略写全
- 是否明确区分了参考库与目标库
- 是否把 Vibe Coding 提示词提升为核心交付物
- 是否已设计出真正可执行的提示词模板蓝图
- 是否明确了保留、简化、放弃三类边界

## 下一步

用户批准后，再调用 `execute-clone-writing`。
