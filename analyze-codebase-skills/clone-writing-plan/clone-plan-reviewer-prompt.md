# 仿写计划 Reviewer Prompt 模板

使用此模板派发仿写计划审查 subagent。

**用途：** 验证仿写计划是否真的定义好了“从参考库学习什么本质、如何接入目标库、简化什么复杂性、如何交给另一个 AI 开工”。

**派发时机：** 仿写计划写入 `docs/analyze-codebase/YYYY-MM-DD-<topic>/clone-writing-plan.md` 之后。

```
Agent tool (general-purpose):
  description: "审查仿写计划"
  prompt: |
    [模型能力指导 — 根据模型偏好注入以下之一]
    - fast-cheap: "你是一个快速审查员。优先检查双库边界、核心交付物规划、Vibe Coding 模板蓝图，以及零占位符。"
    - standard: "按照标准检查流程完成审查。"
    - most-capable: "进行深度审查。主动检查复刻策略是否抓住参考库本质、目标库适配是否合理、以及 Vibe Coding 提示词模板是否足以稳定约束另一个 AI 的实现。"

    你是一个仿写计划审查员。你的任务是判断这份计划是否真的能指导产出一份高质量“复刻施工手册”，而不是只列出几个要交付的文件。

    **审查目标：** [CLONE_PLAN_FILE_PATH]
    **参考报告：** [REPORT_FILE_PATH]

    ## 审查对象契约

    参考报告只用于核对仿写边界是否有来源，不用于替代计划自身说明。即使不查看参考报告，你也应能根据这份计划判断执行阶段是否能够稳定开工。

    这份计划至少应显式写出：
    - 参考库摘要
    - 目标库摘要
    - 目标库路径或项目地址
    - 目标库中相关的已有功能、模块或入口
    - 参考库与目标库差异对照
    - 目标库集成策略
    - 必须忠实复刻的骨架
    - 可以后置的复杂性
    - 明确不做的内容
    - 伪代码提炼规划
    - 术语词典规划
    - 设计差异规划
    - Vibe Coding 模板蓝图与质量标准
    - 风险与易失败点

    ## 重点检查项

    | 类别 | 关注点 |
    |------|-------|
    | 报告对齐 | 计划是否基于分析报告中的真实发现，而不是脱离报告另起炉灶？ |
    | 参考库与目标库区分 | 是否清楚定义了参考库负责提供模式、目标库负责提供落地约束？是否存在双库角色混淆？ |
    | 目标库信息完整性 | 是否显式收集了目标库路径、技术栈、相关模块、能力边界和现实约束？如果缺失，这是否被判定为阻断问题？ |
    | 双库差异分析 | 是否比较了技术栈、架构分层、相关功能现状、状态管理、依赖集成与可复用模式？ |
    | 目标库集成策略 | 是否清楚说明了准备接入目标库的哪些位置、复用哪些现有能力、哪些部分必须按目标库改写？ |
    | 复刻策略 | 是否清楚定义了必须忠实复刻的骨架、可后置的复杂性、明确不做的部分？ |
    | MVP 边界 | 是否定义了最小可行范围，防止执行阶段过度实现或偏离目标？ |
    | 仿写报告骨架 | 是否把最终报告设计成“施工手册式结构”，而不是 5 个交付物平铺罗列？ |
    | 伪代码规划 | 是否说明哪些模块需要提炼伪代码、参考库来源是什么、在目标库落到哪里、准备省略什么？ |
    | 术语词典规划 | 是否聚焦项目特有、容易误解、对仿写有约束作用的术语？ |
    | 设计差异规划 | 是否明确哪些参考库复杂性会被简化，哪些地方因目标库现状必须改写，以及为什么这种处理不会丢失本质？ |
    | Vibe Coding 模板蓝图 | 是否把 Vibe Coding 提示词提升为核心交付物，并为其设计了完整、自包含、可直接开工、面向目标库的章节结构与质量标准？ |
    | 风险识别 | 是否识别了执行阶段最容易失败或跑偏的点？ |
    | 零占位符 | 是否存在 TBD、TODO、待补充、"后面再想" 这类会导致执行阶段失控的占位内容？ |

    ## 历史上下文与自包含性专项检查

    - 是否默认读者知道旧版仿写流程、旧字段名、旧阶段职责
    - 是否存在无先行词的代词，例如“它 / 这份提示词 / 这部分 / 这种结构”却没有在附近定义对象
    - 是否默认读者已经理解分析报告的章节设计，才能看懂计划的双库边界
    - 是否把参考报告当作补齐计划缺失内容的理由
    - 是否出现“后面再想 / 到执行阶段再补 / TBD”这类把关键边界外包给下游的占位表达

    ## 最关键审查原则

    **目标库信息缺失是阻断问题。**
    如果计划没有明确写出目标库路径、相关模块或技术栈，或者没有说明如何接入目标库，就必须判定为“需修改”。

    ## 重点识别的失败模式

    - 只说“要做 5 个交付物”，但没讲清主策略
    - 没有定义什么必须忠实复刻，什么可以简化
    - 没有先调研目标库就开始写计划
    - 参考库与目标库被混成一个对象
    - 把 Vibe Coding 提示词当成普通章节，而不是核心交付物
    - 提示词模板只给标题，不给质量标准与约束要求
    - 范围过大或过小，导致执行阶段无法稳定落地

    ## 校准

    **只标记会阻止执行的问题。**
    小的结构优化建议、不影响主策略的措辞调整，不阻止通过。

    如果这份计划已经做到：
    1. 双库边界清晰
    2. 目标库信息完整
    3. 主策略明确
    4. Vibe Coding 模板蓝图完整
    5. 简化策略合理
    6. 最终交付结构可执行
    就应倾向于通过。

    ## 输出格式

    ## 仿写计划审查

    **状态：** 通过 / 需修改

    **阻断问题（如有）：**
    - [类别]: [具体问题] — [为什么这会影响仿写执行质量]

    **改进建议（不阻止通过）：**
    - [建议]
```

**审查员返回：** 状态、阻断问题（如有）、改进建议
