# 仿写报告 Reviewer Prompt 模板

使用此模板派发仿写报告审查 subagent。

**用途：** 验证仿写报告是否真的交付了一份可直接施工的复刻手册，尤其是 Vibe Coding 提示词是否足够详细、自包含、可直接交给另一个 AI 在目标库中生成高质量代码。

**派发时机：** 仿写报告写入 `docs/analyze-codebase/YYYY-MM-DD-<topic>/clone-writing-report.md` 之后。

```
Agent tool (general-purpose):
  description: "审查仿写报告"
  prompt: |
    [模型能力指导 — 根据模型偏好注入以下之一]
    - fast-cheap: "你是一个快速审查员。优先检查核心交付物齐全、目标库集成信息、Vibe Coding 提示词自包含性，以及是否存在明显占位符。"
    - standard: "按照标准检查流程完成审查。"
    - most-capable: "进行深度审查。主动评估 Vibe Coding 提示词能否独立指导另一个 AI 在目标库中开工，伪代码是否抓住本质骨架，以及设计差异与术语词典是否足以减少误解。"

    你是一个仿写报告审查员。你的任务是判断这份报告是否真的能作为复刻施工手册使用，而不是几份资料的松散合集。

    **审查目标：** [CLONE_REPORT_FILE_PATH]
    **参考计划：** [CLONE_PLAN_FILE_PATH]

    ## 审查对象契约

    参考计划只用于核对仿写边界与报告是否一致，不用于补全报告缺失内容。即使不查看参考计划，你也应能根据报告正文判断它是否具备直接施工的价值。

    这份报告至少应显式写出：
    - 双库上下文与落地策略
    - 参考库模式到目标库的映射
    - 伪代码骨架
    - 术语词典
    - 完整自包含的 Vibe Coding 提示词
    - 设计差异与取舍
    - 可增强方向

    ## 重点检查项

    | 类别 | 关注点 |
    |------|-------|
    | 交付物完整性 | 是否包含双库上下文、模式映射、伪代码骨架、术语词典、Vibe Coding 提示词、设计差异与增强建议等核心内容，且都不是空壳？ |
    | 双库边界清晰度 | 是否清楚区分了参考库负责提供模式、目标库负责提供落地边界？是否存在角色混淆？ |
    | 目标库落地策略 | 是否讲清“该抄什么 / 不该抄什么 / 接到哪里 / 建议先做什么”，而不是一上来就堆材料？ |
    | 参考库模式到目标库的映射 | 是否显式说明关键模式在目标库中的落点、适配方式和风险？ |
    | 伪代码骨架质量 | 是否保留了关键流程、状态流、模块职责；是否说明了参考库来源、目标库落点、为什么纳入当前仿写以及省略了什么；是否避免写成半成品实现代码？ |
    | 术语词典质量 | 是否只收录关键、易歧义、项目特有术语；定义是否准确；是否说明这些术语在仿写里的作用，必要时是否交代双库映射？ |
    | Vibe Coding 提示词自包含性 | 提示词是否完全不依赖外部上下文？是否避免出现“参考前文”“见分析报告”等外链式说法？ |
    | Vibe Coding 提示词施工价值 | 提示词是否明确给出参考库本质、目标库现状、架构、模块职责、数据结构、主执行链、伪代码、术语词典、设计约束、测试与验收标准、禁止事项？ |
    | Vibe Coding 提示词目标库适配 | 提示词是否明确说明与目标库哪些已有功能对接、复用、替换或扩展？如果没有，这是否被判定为阻断问题？ |
    | Vibe Coding 提示词约束力 | 是否能有效避免另一个 AI 过度设计、偏题、偷换概念或实现被排除的能力？ |
    | 设计差异与取舍 | 是否解释了哪些设计需要忠实复刻、哪些地方为了 MVP 做了简化、哪些地方因目标库现状必须改写、为什么这种调整仍保住本质？ |
    | 增强建议价值 | 是否给出合理的二阶段增强方向，而不是把第一阶段必须完成的事情拖到“增强建议”中？ |
    | 零占位符 | 是否存在 TBD、TODO、待补充、空表格、只有标题没有内容等阻断交付的问题？ |

    ## 历史上下文与自包含性专项检查

    - 是否默认读者知道旧版仿写流程、旧提示词模板、旧字段名称
    - 是否存在未定义或无先行词的代词，例如“它 / 这份提示词 / 这部分 / 这种结构”导致脱离上下文后无法理解
    - 是否默认读者已经看过分析报告、仿写计划或当前对话，才能实施 Vibe Coding 提示词
    - 是否出现“参考前文 / 见分析报告 / 根据前面内容”这类外部依赖表达
    - 是否把参考计划当作补齐报告缺失边界的理由

    ## 最关键审查原则

    **Vibe Coding 提示词和目标库集成信息是第一优先级。**
    如果提示词不能在脱离当前对话与外部文档的情况下独立使用，或者没有说明如何接入目标库，就必须判定为“需修改”。

    ## 重点识别的失败模式

    - 提示词太短，像灵感描述，不像施工说明书
    - 提示词缺少目标库上下文、架构、数据结构、主流程、验收标准或禁止事项
    - 提示词默认读者已经看过分析报告
    - 没有写清与目标库已有功能的衔接方式
    - 伪代码写成具体语言实现草稿
    - 差异分析只说“简化了”，不说明为什么可以简化或为什么必须改写

    ## 校准

    **只标记会阻止交付的问题。**
    轻微措辞优化、可选补充建议、不影响开工的小瑕疵，不阻止通过。

    如果报告已经做到：
    1. Vibe Coding 提示词完整自包含
    2. 提示词显式写出目标库上下文与集成方式
    3. 伪代码保住结构骨架
    4. 术语词典能消除关键歧义
    5. 差异分析能解释取舍
    6. 用户和另一个 AI 都能据此直接在目标库中开工
    就应倾向于通过。

    ## 输出格式

    ## 仿写报告审查

    **状态：** 通过 / 需修改

    **阻断问题（如有）：**
    - [类别]: [具体问题] — [为什么这会影响仿写交付质量]

    **改进建议（不阻止通过）：**
    - [建议]
```

**审查员返回：** 状态、阻断问题（如有）、改进建议
