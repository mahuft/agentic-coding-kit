# 执行仿写计划，生成“可直接施工”的仿写报告

你的任务是把参考库分析结果、目标库现实约束和仿写计划转化为一份真正能被使用的复刻手册。最关键的交付物是 **Vibe Coding 提示词**：它必须详细、准确、自包含，把它发给另一个 AI 后，对方应能直接开始实现并产出高质量代码。

<HARD-GATE>
- 禁止输出真正的项目实现代码
- 允许输出伪代码、结构化模板、实施说明、测试标准
- Vibe Coding 提示词必须完整自包含，不允许依赖“参考分析报告 / 参考计划 / 参考上文”
- 如果 `clone-writing-plan.md` 没有显式写清目标库路径、相关模块、技术栈或集成边界，必须先回到计划阶段补齐
</HARD-GATE>

## 术语约定与阶段契约

- **参考库**：指此前被分析和学习的代码库，是本次仿写的模式来源。
- **目标库**：指本次仿写真正要落地的本地项目，是 Vibe Coding 提示词和施工建议必须服务的对象。
- **本次仿写目标范围**：指已在 `clone-writing-plan.md` 中显式定义的复刻边界、简化边界、明确不做项，以及目标库集成边界。
- **当前阶段产物**：指本阶段唯一要生成的 `clone-writing-report.md`。
- **输入**：一份已获用户批准的 `clone-writing-plan.md`，以及前序分析阶段已显式沉淀到文档中的参考库与目标库信息。
- **允许动作**：把计划中的复刻策略转化为仿写报告，必要时对参考库或目标库做少量精确定位阅读，生成伪代码、术语词典、设计差异分析、增强建议，以及完整自包含的 Vibe Coding 提示词。
- **禁止动作**：输出真正的项目实现代码、依赖外部上下文完成核心交付、让执行结果依赖作者脑内的隐含结构、忽略目标库集成约束。
- **输出**：一份可单独交付的 `clone-writing-report.md`，其中 Vibe Coding 提示词必须在脱离当前对话与外部文档的情况下独立使用，并显式说明如何接入目标库。
- **交接规则**：reviewer 只能依据本报告中显式写出的内容做判断；参考计划只用于核对一致性，不用于补全缺失内容。

## 产出目标

最终的 `clone-writing-report.md` 应该像一份“复刻施工包”，帮助用户和另一个 AI 回答：

1. 参考库最值得抄的骨架是什么
2. 目标库当前具备哪些相关能力与约束
3. 本次仿写目标范围要忠实复刻什么、简化什么、接到哪里
4. 最关键的数据流和执行链是什么
5. 如果我要让另一个 AI 实现当前仿写对象，我该怎么准确下指令

## 最终文件结构

在进入模板之前，先明确报告契约：

- `clone-writing-report.md` 必须能在不查看 `clone-writing-plan.md`、`analysis-report.md` 或当前对话的情况下被单独理解。
- 报告必须显式覆盖 **双库上下文与落地策略、参考库模式到目标库的映射、伪代码骨架、术语词典、Vibe Coding 提示词、设计差异与取舍、可增强方向** 七类核心交付物。
- **Vibe Coding 提示词** 不得出现“参考上文 / 参考分析报告 / 见前文 / 根据前面内容”这类外部依赖句式。
- **目标库集成策略** 必须在正文与提示词中同时出现，不能只写在一个位置。
- 如果某个模块、术语或设计差异暂时没有内容，必须写出“暂无 + 原因”，不能留空标题或空表格。

## 第一部分：先讲双库上下文与仿写策略

不要一上来就扔伪代码。先说明：

- 参考库中最值得学习的“架构骨架 / 主流程 / 状态流”是什么
- 目标库当前有哪些现成模块、入口、状态层或基础设施可复用
- 哪些工程复杂性被刻意延后
- 如果让一个 AI 来做，推荐先实现哪条主路径、接到目标库哪里

## 第二部分：伪代码骨架

### 要求
- 使用结构化自然语言
- 保留关键流程、状态转换、数据流
- 省略与当前复刻目标无关的工程细节
- 每段伪代码都要说明“为什么纳入当前仿写”
- 每段伪代码都要说明在目标库中的落点

### 每个模块都必须包含
- 参考库来源
- 目标库落点
- 纳入理由
- 核心逻辑概括
- 伪代码
- 省略内容及原因

## 第三部分：术语词典

词典的使命是：让 Vibe Coding 提示词里的每个关键名词都不会歧义。

### 词典条目要求
- 精确定义，不说套话
- 只收录非通用术语、项目特有概念、容易误解的重载词
- 每个术语必须说明它在仿写里的作用
- 必要时说明术语在参考库与目标库中的映射关系

## 第四部分：Vibe Coding 提示词（最重要）

### 这是整个报告里最核心的交付物

你必须生成一份可直接复制给另一个 AI 的完整提示词。它必须：

- **完全自包含**：不依赖外部文档
- **足够详细**：让另一个 AI 直接知道做什么、先做什么、不要做什么
- **约束清晰**：防止 AI 过度设计、偏题、偷换概念
- **能产出代码**：不是灵感提示，而是真正的实施说明书
- **面向目标库落地**：明确说明该能力如何接入目标库已有结构

### 生成这份提示词时的额外要求

- 所有必要上下文都必须内联
- 不要出现“参考分析报告”“见上文”“根据前面内容”之类依赖外部上下文的话
- 必须体现：
  - 双库背景
  - 目标库上下文
  - 范围
  - 架构
  - 数据结构
  - 主流程
  - 伪代码
  - 术语
  - 验收标准
- 允许写得详细，重点是降低误解

## 第五部分：设计差异与取舍

这部分要解释：
- 哪些地方需要忠实复刻参考库设计
- 哪些地方因为目标库现状必须改写
- 哪些地方为了 MVP 故意简化
- 为什么这些调整仍然保住了本质

## 第六部分：可增强方向

告诉用户：
- 当 MVP 跑通后，下一步最值得补什么
- 哪些复杂性应该在第二阶段再引入
- 哪些优化不应在第一版就上

## 质量标准

一份合格的仿写报告必须满足：

- 用户能理解“该抄什么，不该抄什么，接到哪里”
- 伪代码能帮助另一个 AI 理解结构骨架
- 术语词典能消除关键歧义
- Vibe Coding 提示词拿去就能开工，不依赖任何额外上下文
- 差异分析能说明每次简化或改写的理由
- 目标库集成策略在正文和提示词中都明确可见

## Red Flags

- Vibe Coding 提示词过短，只是泛泛描述
- 需要结合外部报告才能理解提示词
- 伪代码变成了某种语言的半成品实现
- 没有交代保留/简化/放弃三类边界
- 没有交代与目标库已有功能的衔接方式
- 设计差异只说“简化了”，不说为什么可以简化或为什么必须改写

## 自审要求

reviewer 必须重点检查：
- Vibe Coding 提示词是否真的自包含
- 是否足够详细，能直接指导另一个 AI 实现目标库代码
- 是否约束清晰，避免实现跑偏
- 是否真正体现了参考库中提炼出的本质骨架
- 是否明确说明了如何接入目标库
