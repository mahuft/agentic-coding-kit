# 生成“编码任务计划说明书”

你的任务是把 `code-imitation-spec.md` 转化为一份真正能被新的 Coding Agent 直接执行的 `coding-task-plan.md`。这不是教学报告，也不是资料拼盘，而是一份完整、自包含、可直接发送给另一个 AI 的 Agentic Coding 提示词。

<HARD-GATE>
- 禁止输出真正的项目实现代码
- 允许输出伪代码、接口契约、数据结构、流程说明、实施顺序、测试与验收标准
- `coding-task-plan.md` 必须完全自包含，不允许依赖 `analysis-report.md`、`code-imitation-spec.md` 或当前对话补足核心上下文
- `coding-task-plan.md` 不得使用“参考库”“目标库”这类只在当前工作流中成立的元术语；必须直接写出具体项目路径、模块名、数据结构、流程与约束
- 采用单文件分章节门控：每完成一章都必须等待用户批准，未获批准不得继续下一章
</HARD-GATE>

## 全局提问原则

本工作流要求：**尽可能在规划阶段提出问题，好过于真正执行后再发现问题。**

在本阶段必须遵守以下原则：

1. 在写某章之前，先检查当前规格是否足以支持这一章可靠落笔。
2. 若发现会影响方向、接口、边界、优先级、验收口径或具体代码落点的高影响不确定性，必须主动提问。
3. 若存在更优实现路线或更稳妥的落地方案，可以先提出方案建议与推荐，再继续写章节。
4. 提问应一次一个、聚焦决策，优先消除会导致后续编码质量下降或返工的问题。
5. 若可通过当前项目的轻量扫描确认事实，先探索；若属于用户意图或方案取舍，再问用户。

## 术语约定与阶段契约

- **参考库**：指被学习与提炼模式的来源，只在本技能内部用于整理知识；这些信息在最终文档中必须被改写成自包含事实，不能以“参考库”标签暴露给新的 AI。
- **目标库**：指本次编码任务真正要落地的代码库，只在本技能内部用于组织信息；最终文档必须直接写出具体仓库路径、模块名、约束与上下文，而不是使用“目标库”标签。
- **当前阶段产物**：指本阶段唯一要生成的 `coding-task-plan.md`。
- **输入**：一份已获用户批准的 `code-imitation-spec.md`，以及当前项目中可通过最小必要扫描确认的事实。
- **允许动作**：把规格中的复刻策略转化为可直接发送给新 AI 的章节化编码任务计划，必要时做少量精确定位阅读，提问澄清高影响问题，提出更优落地方案，生成伪代码、接口契约、数据结构、流程说明、工程约束与测试标准。
- **禁止动作**：输出真正的项目实现代码、依赖外部文档补全关键信息、忽略当前项目集成边界、绕过章节级批准。
- **输出**：一份可单独交付的 `coding-task-plan.md`，整份文档就是可直接发送给新 AI 的完整提示词。
- **交接规则**：reviewer 只能依据本计划中显式写出的内容做判断；`code-imitation-spec.md` 只用于核对一致性，不用于补齐缺失信息。

## 产出目标

最终的 `coding-task-plan.md` 应帮助新的 Coding Agent 明确回答：

1. 要在当前项目中实现什么
2. 为什么做、做多少、不做什么
3. 应接到哪些现有模块或入口，复用或改写哪些现有能力
4. 关键模块、数据结构、接口契约、主流程与边界流程是什么
5. 实施顺序、测试要求与验收标准是什么
6. 哪些设计不能偏离，哪些复杂性暂时不要做

## 写作原则

- **项目内联**：整份文档必须直接写出项目路径、模块名、文件落点、接口约束与上下文，不得用“目标库”“参考库”代称。
- **自包含**：所有必要上下文都必须内联，不得出现“见参考报告”“见前文”“参考上面章节”这类外部依赖表达。
- **执行导向**：减少教学解释、减少重复背景、减少平行罗列，强化职责、契约、流程、约束与验收。
- **细节充分**：如果缺少详细约束会让新的 AI 产出低质量代码，就必须把这些约束写进文档，而不是追求简短。
- **先澄清再落笔**：发现高影响不确定性时先提问或提方案，不得硬写。

## 章节化生成流程

1. 读取已获用户批准的 `code-imitation-spec.md`
2. 只提取并消费下列显式字段：
   - 项目路径与相关模块/入口
   - 双库差异与推荐方案
   - 项目集成策略
   - MVP 范围、非目标与成功标准
   - 必须保留的本质与允许简化项
   - 风险、依赖、待确认事项
3. 如果规格缺少上述关键字段，先回到规格阶段补齐，不要自行脑补
4. 按章节顺序生成 `coding-task-plan.md`
5. 每章写入同一个文件后，暂停并等待用户批准
6. 若某章开写前发现重大歧义，先提问或提出更优方案
7. 全部章节通过后，派发 reviewer 做全文自审

## `coding-task-plan.md` 文档结构与章节要求

`coding-task-plan.md` 的章节数可以随复杂度扩展，但至少要覆盖下面这些信息块；如果某个主题复杂到一章无法写清，可以拆成两个连续章节。不要同时维护一份“章节说明”和一份重复模板，最终文档直接按这些章节要求落地。

### 章节 1：任务定义与项目上下文
必须写清：
- 当前项目的绝对路径或项目标识
- 本次要实现的能力与最终交付物
- 与本次任务最相关的现有模块、入口、基础设施与约束
- 本阶段明确不做的事情

### 章节 2：范围、非目标与成功标准
必须写清：
- 第一版必须实现什么
- 明确排除什么
- 每个关键能力的完成判定标准
- 哪些失败结果算未完成

### 章节 3：现有代码上下文与复用策略
必须写清：
- 当前项目里已经存在什么能力可以复用
- 哪些模块需要扩展，哪些模块需要新增
- 哪些现有行为不能破坏
- 相关文件、目录、类型、接口的具体落点

### 章节 4：目标设计与模块职责
必须写清：
- 建议模块划分与职责边界
- 模块之间的调用关系、状态归属与副作用边界
- 每个核心模块在当前项目中的具体落点
- 哪些设计约束绝不能偏离

### 章节 5：数据结构、接口契约与状态模型
必须写清：
- 关键数据结构、字段、含义、生命周期
- 核心接口的输入、输出、约束与错误语义
- 状态转换规则与关键不变量
- 新旧结构之间的映射关系（如果有）

### 章节 6：主流程、边界流程与伪代码
必须写清：
- 一条主执行链如何完整跑通
- 关键边界流程、错误路径、回退路径如何处理
- 结构化伪代码骨架
- 伪代码对应的模块落点与责任划分

### 章节 7：实施顺序、测试与验收
必须写清：
- 推荐实施顺序，按依赖关系分步推进
- 每一步应该先修改哪些文件或模块
- 最低测试要求、关键场景与验收方式
- 禁止事项与常见跑偏方式

## `coding-task-plan.md` 生成约束

生成最终文档时，必须同时满足以下要求：

1. 标题、章节名和正文都应直接面向新的 AI，不要写“目标库上下文”“参考库做法”这类工作流内部标签。
2. 需要表达参考来源时，只能把有用事实改写成自包含说明，例如“沿用现有事件流状态推进方式”，不能写“参考库里是这样做的”。
3. 需要表达当前项目上下文时，必须直接写项目路径、模块名、文件路径、类型名、接口名，而不是写“目标库已有能力”。
4. 文档必须足够详细：如果新的 AI 仅凭当前文档仍无法判断模块边界、数据契约、流程顺序或验收标准，说明内容还不够。
5. 文档可以很长；优先保证精确与完整，避免为了简短牺牲约束力。

## 章节门控要求

- 每章完成后都必须请求用户批准。
- 未获批准不得生成下一章。
- 若用户要求修改，先更新当前章，再重新等待批准。
- 全文通过后才能派发全文 reviewer 自审。

## 自审要求

reviewer 必须支持两级审查：

1. **章节级审查**：当前章节是否足够驱动实现，是否还存在应当前置澄清的问题。
2. **全文级审查**：整份提示词是否自包含、是否完全服务当前项目、是否无外部依赖、是否存在更优方案被忽略。

## Red Flags

- 整份文档仍然像教学报告，而不是执行说明
- 仍依赖上游文档补充关键上下文
- 文档继续使用“参考库”“目标库”这类新 AI 无法理解的工作流元术语
- 没有说明具体模块、文件、接口和代码落点
- 关键数据结构、接口契约、主流程或验收标准缺失
- 明知有重大歧义，却继续硬写下一章
